### 2.2 后端启动链路已修正
已修改：

- `backend/app/services/agent_service.py`
- `backend/app/main.py`

做过的修正：

#### A. 把 AI 相关初始化改成懒加载
之前的问题：
- `agent_service.py` 在模块导入时就会初始化 `SentenceTransformer`、embedding、vector store。
- 这会导致后端 import 阶段就变重，表现为启动慢、首次行为不透明、出问题时很难判断卡在哪里。

现在的处理：
- `ChatOpenAI` 改为通过 `get_chat_llm()` 获取。
- embedding 改为通过 `get_embeddings()` 获取。
- Chroma 改为通过 `get_vector_store()` 获取。
- `AgentService` 中的 `vector_store` 也改为真正使用时才初始化。
- `CPUEmbeddingFunction` 里把 `SentenceTransformer` 的 import 下沉到类初始化内部，进一步避免模块导入阶段就触发大依赖加载。

效果：
- 后端可以更快进入 ready 状态。
- embedding/向量库的加载被推迟到第一次真正用到 AI/RAG 时。

#### B. RAG collection 命名与索引预热已补上
现在的处理：
- Chroma collection 名会根据 `EMBEDDING_PROVIDER` 和 `EMBEDDING_MODEL` 自动生成，避免不同 embedding 配置共用同一个 collection。
- 后端启动后会执行 `AgentService.ensure_bootstrap_index()`：
  - 如果当前 collection 为空，就把数据库里已有的有效文档重新索引进去。
  - 如果 collection 已经有数据，则跳过预热。
- 相关日志里会带上 `collection_name`，方便后续排查 RAG 问题。

这意味着：
- 服务器重启后，如果向量库为空，不需要人工手动重建所有索引。
- 更换 embedding provider / model 后，新的 collection 会自动隔离，不容易和旧数据混用。

#### C. 增加健康检查接口
已新增：
- `GET /health`

用途：
- 快速判断服务进程是否已 ready。

#### D. 数据库启动失败不再静默吞掉
之前的问题：
- `main.py` 的 startup 会记录异常但不抛出，容易出现“进程在，但服务不完整”的假启动状态。

现在的处理：
- 数据库初始化失败会明确抛出，启动直接失败。
- Redis 仍允许降级到内存缓存，不阻断服务。

### 5.2 启动阶段会做一次 RAG 预热
当前 `main.py` 已经在 startup 里加入：

- 数据库初始化完成后，创建一个短生命周期 session
- 调用 `AgentService.ensure_bootstrap_index()`

行为特点：
- 如果当前 collection 为空，会把数据库中已有有效文档重新建索引
- 如果 collection 已有数据，则不会重复全量重建
- 这会让“第一次 RAG 请求”的成功率更高，但也意味着某些情况下启动阶段会比纯 API 服务稍慢一些

这属于当前设计的一部分，不要误判成异常。

这不是项目代码逻辑问题，而是当前系统环境不完整。

## 7. 推荐启动方式

### 后端
```bash
cd /root/Velo/backend
DATA_DIR=/root/Velo/backend/data \
POSTGRES_SERVER=sqlite \
LLM_PROVIDER=ollama \
LLM_BASE_URL=http://localhost:11434/v1 \
COMPLETION_MODEL=qwen2.5-coder:14b \
CHAT_MODEL=qwen2.5:7b-instruct \
EMBEDDING_PROVIDER=huggingface \
EMBEDDING_MODEL=BAAI/bge-small-zh-v1.5 \
OPENAI_API_KEY=ollama \
/root/Velo/.venv/bin/python -m uvicorn app.main:app --host 127.0.0.1 --port 8000
```

如果 8000 被占用，可改成：
```bash
--port 8001
```

### 前端
```bash
cd /root/Velo/frontend
npm run dev -- --host 127.0.0.1 --port 5173
```

如果端口被占用，Vite 会自动顺延到 5174 / 5175 / 5176。
