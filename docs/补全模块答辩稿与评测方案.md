# 补全模块答辩稿与评测方案

## 1. 3 分钟答辩稿

我的系统里有一个文档补全模块，表面上看它只是“AI 自动续写”，但我在实现时把它当成一个更严格的研究问题来处理。因为普通的大模型补全很容易出现三个问题: 第一是结构漂移，比如明明在段落里，却突然生成一个新的列表项；第二是模板复读，比如反复输出“还包括”“如下”；第三是空泛套话，导致补全虽然通顺，但没有信息增量。

针对这个问题，我没有继续单纯调 Prompt，而是把补全任务重新建模为“面向中文 Markdown 写作的带约束中缀生成任务”。系统输入不是一整段文章，而是光标前文、后文、语言类型和触发模式。然后系统先对当前光标环境做状态分类，判断它属于段落、列表项、标题还是引用。不同状态下，允许生成的文本结构是不一样的。

在此基础上，我设计了三层约束。第一层是策略约束，也就是按上下文动态构造 Prompt 和示例。第二层是解码约束，包括 stop 序列、单行输出和长度上限。第三层是后验审核，也就是用确定性规则检查模型输出是否合法，如果出现未授权列表、新的 Markdown 块、模板引导句或者复读，就直接拒绝，并触发一次重试。

这样做的核心价值是把“生成能力”和“结构正确性”分开了。模型负责提出候选，而系统策略层负责决定什么样的候选可以被接受。相比传统只靠提示词的方法，这种做法更可解释、更稳定，也更容易测试。

在验证方面，我做了三类测试。第一类是单元测试，验证上下文分类是否正确。第二类是规则测试，验证像“王安石变法的经济政策体系还包括”这样的错误模板会被拒绝。第三类是端到端验证，确认在真实 Markdown 场景中，段落态补全不会无故起列表，而显式列表项环境下又可以正常补全文本。

所以，这部分工作的重点不只是“把补全做出来”，而是把它提升成一个有明确问题定义、有方法设计、有验证闭环的研究型模块。

## 2. 5 分钟扩展版答辩稿

### 2.1 研究背景

在写作型系统中，补全功能通常直接调用大模型做续写。但在真实文档编辑场景下，这种做法会有明显缺陷，因为编辑器补全不是开放式问答，而是局部、即时、受结构约束的中缀生成任务。

### 2.2 主要问题

我在系统测试中发现三类典型错误:

- 结构错误: 在段落内部无故新建列表或标题
- 模板错误: 固定引导句反复出现，例如“还包括：”“如下：”
- 语义错误: 补文空泛、重复、与局部上下文不紧密

### 2.3 方法设计

我把解决方案设计为四个阶段:

1. 上下文感知  
基于 `prefix` 当前行，识别当前状态是段落、列表项、标题还是引用。

2. 策略条件化生成  
根据状态构造不同 Prompt，并显式写明允许和禁止的结构。

3. 解码约束  
通过 stop 序列、长度限制和单行输出，将候选控制在编辑器可接受范围内。

4. 后验审核  
对模型输出做确定性判别，拒绝模板复读、结构漂移和无效补文。

### 2.4 系统贡献

这个模块的贡献不在于“调通了一个 API”，而在于:

- 把补全问题形式化
- 构建了一个可解释的状态分类机制
- 通过策略层解耦业务规则和模型调用
- 形成了可测试、可验证的闭环系统

### 2.5 工程与研究结合点

对工程来说，它提高了补全稳定性。  
对研究来说，它展示了如何把生成式 AI 模块从“经验式 Prompt 调参”提升为“约束驱动的系统设计”。

## 3. 评测方案

### 3.1 评测目标

评测不只看“像不像人写的”，还要看“结构是否合法、是否有信息增量、是否能被用户接受”。

### 3.2 建议指标

#### 结构类指标

- `Structure Violation Rate, SVR`
  定义: 输出中出现未授权 Markdown 结构的比例
- `Lead-in Error Rate, LER`
  定义: 输出包含“还包括：”“如下：”等无效引导句的比例
- `Suffix Echo Rate, SER`
  定义: 输出与后文发生回声重叠的比例

#### 内容类指标

- `Generic Phrase Rate, GPR`
  定义: 输出包含空泛套话的比例
- `Local Relevance Score, LRS`
  定义: 人工标注补文与局部上下文是否紧密相关，可采用 1 到 5 分 Likert 量表
- `Completion Utility Score, CUS`
  定义: 人工标注该补全是否真正节省写作成本

#### 交互类指标

- `Acceptance Rate, AR`
  定义: 用户按 Tab 接受补全的比例
- `Dismissal Rate, DR`
  定义: 用户按 Esc 或继续输入覆盖 ghost text 的比例
- `Latency`
  定义: 从触发补全到前端展示 ghost text 的耗时

### 3.3 数据构建建议

可以从自己的中文文档中抽取 100 到 300 个补全点，构造成数据集:

- 段落内部补全点
- 列表项补全点
- 标题补全点
- 引用补全点

每个样本至少记录:

- `prefix`
- `suffix`
- `language`
- `expected block type`
- `human reference`
- `accept / reject label`

### 3.4 对比实验

建议做 4 组对比:

1. `Baseline`
   只用统一 Prompt，无结构分类、无后验审核

2. `Baseline + Context Classification`
   只加状态识别和差异化 Prompt

3. `Baseline + Context Classification + Decoding Constraints`
   再加 stop 和长度约束

4. `Full Model`
   加入后验审核与拒绝重试

这样可以形成一个较标准的消融实验结构。

### 3.5 预期结论表达

你可以在论文或答辩中这样表达:

> 实验结果表明，仅依赖统一提示词的补全方案容易产生结构漂移和模板复读；加入上下文状态分类后，局部相关性显著增强；进一步加入确定性后验审核后，结构违规率和无效引导句比例明显下降，说明约束驱动的生成框架在中文 Markdown 写作补全任务中具有更好的稳定性。

## 4. 消融实验写法模板

### 4.1 消融目的

验证各个模块对最终性能的贡献，而不是只证明“整体能运行”。

### 4.2 消融变量

- 去掉上下文分类
- 去掉结构化 Prompt
- 去掉 stop 约束
- 去掉后验审核
- 去掉拒绝后重试

### 4.3 消融分析模板

可直接写成:

> 当移除上下文分类模块后，模型更容易在段落态下误生成列表项，说明状态识别对结构稳定性至关重要。  
> 当移除后验审核模块后，模板化引导句和空泛补文显著增加，说明仅依赖模型内在约束不足以保证可用性。  
> 当移除重试机制后，系统面对一次性失败候选的恢复能力下降，整体接受率有所降低。

## 5. 复试老师可能会问的问题

### 5.1 为什么不用纯 Prompt 解决？

答: 因为 Prompt 只能提高概率，不能保证结构合法性。对于编辑器补全这种强约束场景，仅靠 Prompt 不够稳定，所以我增加了确定性策略层。

### 5.2 你的方法和普通聊天生成有什么本质区别？

答: 聊天生成是开放输出，而编辑器补全是局部中缀生成，必须满足结构、长度和位置约束，因此更适合做上下文分类和约束解码。

### 5.3 为什么说它有研究性？

答: 因为它不仅是接口调用，而是对问题进行了形式化建模，并设计了状态分类、条件化生成和后验审核的分层框架，还可以通过消融实验验证各模块贡献。

### 5.4 规则会不会太死？

答: 规则层只负责保证底线正确性，比如不允许段落中无故起列表。它不会替代模型生成，而是和模型形成“候选生成 + 合法性筛选”的协作关系。

### 5.5 后续还能怎么改进？

答: 可以继续引入代码块、表格、公式等更复杂结构；也可以把规则层抽象为可切换 policy profile，甚至引入有限状态自动机或 CFG 约束。

## 6. 最后一句总结

如果需要一句适合答辩收尾的话，可以直接说:

> 这个补全模块的价值不只是提升了写作体验，更重要的是，它把一个看似简单的“AI 续写功能”重构成了一个具有明确问题定义、分层方法设计和可验证实验框架的研究型系统模块。
