## 1. 这份稿子的用途

这份稿子是给老师讲的口语版，不是技术白皮书。

目标有三个：

1. 先把系统说清楚。
2. 再把你的核心贡献说出来。
3. 最后把“为什么这样做有效”讲成老师愿意继续追问的结构。

你可以直接照着讲，也可以把每一节压缩成自己的话。

建议总时长：

1. 开场和定位：40 秒
2. 系统总体结构：1 分钟
3. 索引与检索：2 分钟
4. 证据评分与回答路由：2 分钟
5. 例子和实验结果：1 分 30 秒
6. 收尾：30 秒

## 2. 最稳的开场说法

老师好，我这部分主要做的是本地知识库问答系统里的 RAG 主链路优化。  
我关注的重点不是通用聊天，而是两个更实际的问题：第一，系统能不能把真正相关的文档稳定找出来；第二，当证据不够的时候，它能不能明确拒答，而不是让大模型自由发挥。

我现在这套系统已经不是“向量检索一下，然后把结果直接塞给大模型”那种最基础的 RAG，而是一条比较完整的流水线。  
它会先理解 query 的类型和证据要求，再做向量召回和 BM25 词法召回的混合检索，然后用 rerank 精排，接着通过统一证据评分器判断候选文档到底能不能用，最后再决定是拒答、抽取式回答、结构化短答，还是生成式综述。

我这一部分的核心贡献主要有两块：  
第一块是检索侧的自适应 hybrid 融合和候选池构造；  
第二块是回答前的统一证据评分和显式模式路由。  
简单说，就是把系统从“检索后直接生成”的黑盒，做成了“意图驱动、证据驱动、可解释”的流水线。

## 3. 系统总体结构怎么讲

如果老师让我先整体介绍，我会这样讲：

当前系统前端是一个本地文档编辑器加知识库问答面板，后端用 FastAPI 提供接口，核心数据存储有三层：

1. SQLite 存原始文档。
2. Chroma 存向量索引。
3. Redis 优先做缓存，Redis 不可用时会自动降级到进程内存缓存。

知识库问答的接口是 `/api/v1/agent/chat`。  
前端把用户问题发到后端之后，后端会进入 `RagService.rag_qa` 这一条主链路。

这条主链路按顺序可以概括成：

```text
用户问题
-> QueryIntentBuilder
-> 向量召回 + BM25 召回
-> adaptive hybrid 融合
-> cross-encoder rerank
-> UnifiedEvidenceScorer
-> AnswerModeRouter
-> Extractive / Structured / Generative / NoContext
```

所以我的系统重点不是单独调一个 Prompt，而是把“检索、证据判断、回答方式”都做成了可控制的模块。

## 4. 索引层和入库流程怎么讲

这部分建议你讲得简洁但具体。

我现在的文档不是整篇直接丢进向量库，而是先做结构化切分。  
后端在文档创建或更新后，会通过后台任务触发索引更新。

具体做法是：

1. 先按 Markdown 标题切分。
2. 再用递归切块，参数是 `chunk_size=1000`，`chunk_overlap=200`。
3. 每个 chunk 写入 Chroma 前，会补上 `source` 和 `doc_id` 这两个关键元数据。
4. 我还在入库前加了一个轻量语义锚点，也就是从标题和首段里抽取核心实体，生成类似 `【核心主题：...】` 的前缀，再注入到 chunk 头部。

这样做的原因是：

1. 标题切分能保留文档原来的章节结构。
2. 递归切块能兼顾局部语义和上下文连续性。
3. 元数据保证后面引用来源可追踪。
4. 语义锚点能减少“只是顺带提到某个实体却被误召回”的情况。

当前这套演示知识库里，数据库中有效文档有 4 篇，向量库里一共 7 个 chunk。  
其中三篇较短文档各切成 1 个 chunk，长文档《雾潮镇的档案馆》切成 4 个 chunk。  
这也说明我的系统既考虑小文档，也考虑长文档。

## 5. 查询理解和检索层怎么讲

这一段是你最该讲出技术感的地方。

我没有把 query 原样拿去搜，而是先经过 `QueryIntentBuilder` 做轻量但明确的意图构建。  
这一步主要做三件事：

1. 规范化 query。
2. 判断问题类型。
3. 生成后续检索和防御所需的控制参数。

规范化主要包括：

1. 去掉“请问一下”“帮我看看”这种口语前缀。
2. 去掉句尾语气词。
3. 生成两个 query 视图：
   - `normalized_query`：更接近原问题语义
   - `keyword_query`：更适合 BM25 做词法匹配

然后系统会把问题分成几类，比如：

1. `LOOKUP`，比如只问一个实体名。
2. `FACTOID`，比如问一个具体事实。
3. `RELATION`，比如问两个实体之间的关系。
4. `REASON`，比如问为什么。
5. `SUMMARY` 或 `OVERVIEW`，比如问历史、概况、主要内容。

这一步做完以后，后面检索就不是一刀切了。

我当前用的是双路检索：

1. 一路是向量召回，负责语义相近。
2. 一路是文档级 BM25，负责精确词法命中。

这里我不是简单把两路分数硬加起来，而是先做查询画像，再动态决定 dense 和 lexical 的权重。  
比如短 query、编号 query、关键词特别强的 query，就更依赖词法；  
长 query、抽象 query，就更依赖 dense 语义检索。

在这个基础上，我把多个信号融合成一个 `adaptive_score`，主要包括：

1. dense 归一化分数
2. BM25 归一化分数
3. RRF 排名融合分数
4. query 词覆盖率
5. identifier 一致性

这样做的目标不是直接回答问题，而是先把候选池做干净。  
因为如果候选池一开始就脏，后面 rerank 和生成都很难补救。

## 6. Rerank、证据评分和拒答边界怎么讲

老师如果继续听，一般最关心“你怎么保证答得靠谱”。

我这里做了两层收口。

第一层是 rerank。  
系统会把 hybrid 阶段得到的候选送给 cross-encoder，也就是 `BAAI/bge-reranker-v2-m3` 做精排。  
它的作用是对高质量候选做更深的相关性判断。

但我没有让 rerank 一票否决，因为只看 rerank 也会有两个问题：

1. 它可能忽略检索阶段已经得到的先验信息。
2. 它不能单独解决“文档提到了关键词，但其实不是用户真正想问的主体”这种问题。

所以第二层我做了一个统一证据评分器，也就是 `UnifiedEvidenceScorer`。  
它会把以下信息统一收口：

1. hybrid 得到的 `adaptive_score`
2. rerank 得到的 `rerank_score`
3. query 与标题、正文的 `topic_alignment`
4. 在部分高风险问题上，再额外触发一次 `LLM Judge`

这里我要强调一点：  
Judge 不是全量启用的。  
当前代码里，Judge 主要用于 `STRICT + atomic-span` 这类高风险短实体、地点或属性型 query。  
像关系型、多段综合型问题，当前更依赖统一证据评分和回答路由，而不是强行让 Judge 逐个卡。

统一证据评分器最后会给每个候选一个结果：

1. `usable`
2. `reject_reason`
3. `flags`

也就是说，系统不是“只要检索到一点相关内容就尽量回答”，  
而是先明确判断“这个候选能不能作为证据”，如果不行就拒掉。

这一步就是我系统里拒答边界最核心的地方。

## 7. 回答模式路由怎么讲

有了 usable evidence 之后，系统不会统一走大模型生成，而是进入 `AnswerModeRouter`。

这里有四种模式：

1. `NO_CONTEXT`：没有可用证据，直接拒答。
2. `EXTRACTIVE`：适合短事实、能直接从原文抽句回答的问题。
3. `STRUCTURED`：适合需要整理但不需要长篇综述的问题。
4. `GENERATIVE`：适合概况、历史、总结、多文档综合这类问题。

比如：

1. 问“雾潮镇档案馆的正式名称是什么”，通常会走抽取式。
2. 问“为什么研究者重视 A-17-204 蓝布账册”，更适合结构化或生成式。
3. 问“雾潮镇的历史”这类概况题，就直接走生成式。

这样做的好处是：

1. 事实题尽量短、准、可引用。
2. 综合题再让模型做组织。
3. 资料不足时直接拒答。

所以我的系统不是“一个生成器打天下”，而是“先判断问题，再决定回答形式”。

## 8. 用一个具体例子带老师走一遍

如果老师让我举例，我建议你讲这道题：

“研究者为什么重视编号为 A-17-204 的蓝布账册？”

这道题很适合，因为它能体现多个模块的配合。

我会这样讲：

第一，这个问题会先被识别成 `REASON` 类型，而且因为有编号，所以关键词视图里会保留 `17`、`204`、`蓝布`、`账册` 这些词。  
第二，向量召回和 BM25 都会命中《雾潮镇的档案馆》这篇文档。  
第三，adaptive hybrid 会给这篇文档比较高的先验分，因为它同时满足语义相关、词法覆盖和编号一致。  
第四，rerank 会进一步确认这篇文档确实和问题强相关。  
第五，统一证据评分器会判断这不是一个“单句就能答完”的原子事实题，而是一个需要综合上下文的原因型问题。  
最后，路由器会把它送到结构化或生成式回答，而不是硬抽一句不完整的话。

这样老师就能看到，这个系统的优势不只是“能答”，而是“知道该怎么答”。

## 9. 实验和验证结果怎么讲

这一段一定要讲，因为老师会关心你不是只在本地 demo 上自我感觉良好。

我这套检索方法不仅接进了后端，还做了几组独立验证。

第一组是公开中文检索 benchmark。  
我用了 C-MTEB 里的 `DuRetrieval` 和 `T2Retrieval` 两个数据集，对比了三条链路：

1. 主流方案 A：Dense + Rerank
2. 主流方案 B：Hybrid + RRF + Rerank
3. 我的方案：Adaptive Hybrid + Rerank

结果上，我的方案相对主流方案 B 仍然有正提升。  
比如在 `DuRetrieval` 上：

1. 主流 B 的 `Hit@1` 是 `0.8408`
2. 我的方案是 `0.8558`
3. `nDCG@10` 从 `0.6616` 提到 `0.6858`

在 `T2Retrieval` 上，也有小幅提升：

1. `Hit@1` 从 `0.8450` 到 `0.8475`
2. `nDCG@10` 从 `0.6904` 到 `0.6984`

第二组是消融实验。  
消融结果显示，当前收益最大的两项是：

1. 查询自适应权重
2. 混合候选池

覆盖率奖励有小幅正收益；  
标识符约束在公开数据集上没有表现出稳定正收益，所以我不会把它硬讲成核心创新。

第三组是系统级评测。  
我还把当前系统和 LlamaIndex 方案做了对比，在本地知识库、RGB 数据集和 CRUD-RAG 基准上都做了评测。  
比如在 RGB 上，我的系统准确率大约是 `0.8033`，对比方案大约是 `0.59`；  
在本地知识库上，我的拒答 precision 和 recall 都是满分，说明系统边界控制是有效的。

所以这套方法不是只对一个 demo 有效，而是在检索 benchmark 和系统评测上都有支撑。

## 10. 最后的收尾说法

最后我会这样总结：

总体来说，我这部分工作的核心，不是单独换了一个模型，也不是单独写了一个 Prompt，  
而是把 RAG 从“检索后直接生成”的松散流程，重构成了一条可解释、可控制的知识库问答主链路。

这条主链路的重点是三件事：

1. 通过自适应 hybrid 和 rerank，把候选文档找得更准。
2. 通过统一证据评分，把“能不能答”这件事前移。
3. 通过回答模式路由，让系统根据问题类型选择合适的回答方式，并在证据不足时稳定拒答。

如果用一句话概括我的贡献，那就是：

我把这个 RAG 系统从“能检索、能生成”，推进到了“知道检索什么、知道凭什么回答、也知道什么时候不该回答”。

## 11. 如果老师追问，你可以马上补的三句

### 11.1 如果老师问“你的创新点到底是什么”

你就答：

我的创新点不在单一模型，而在主链路设计。  
算法上主要是 query 自适应的 hybrid 融合和混合候选池；  
工程上主要是统一证据评分器和显式回答路由。

### 11.2 如果老师问“Judge 是不是每次都跑”

你就答：

不是。当前代码里 Judge 是选择性启用的，主要用于高风险、原子证据型 query。  
多段综合型问题更依赖统一证据评分和回答路由，而不是把所有问题都交给 Judge。

### 11.3 如果老师问“你的系统最大的不足是什么”

你就答：

目前还有几个工程上明确可见的改进点：  
比如响应缓存没有跟文档更新联动失效，BM25 词法索引刷新也还可以做得更严谨；  
另外当前接口虽然收的是 message 列表，但后端实际上只取最后一条用户问题，所以现在更准确地说，它是单轮知识库问答，而不是完整会话记忆系统。

## 12. 你答辩时最稳的五句口径

1. 直接说当前接口层依赖的是 `RagService`，真正执行的是 `RagService.rag_qa`。
2. 直接说 Judge 是选择性启用，不是每个问题都会跑。
3. 直接说当前主链路只保留 query 去噪、关键词视图、hybrid 检索、统一证据评分和模式路由，不再额外叠历史增强模块。
4. 直接说当前运行形态是单轮知识库问答，接口虽然接收消息列表，但后端实际只使用最后一条用户消息。
5. 直接说公开消融里收益最明确的是自适应权重和混合候选池，标识符约束目前更适合讲成辅助信号。
