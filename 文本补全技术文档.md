# 文本补全技术文档

## 1. 文档目标

这份文档专门说明当前版本的文本补全技术实现，目标是给后续维护者提供一份可以直接接手的技术参考。

本文不再从仓库历史、故障报告或旧架构文档出发，而是只基于今天这一轮改动后的补全链路来说明：

- 补全为什么这样设计
- 前端如何触发补全
- 后端如何构造上下文和 prompt
- 为什么会出现空补全、泛补全、重复补全
- 现有实现如何做质量控制
- 后面还能往哪些方向继续优化

补全系统的核心目标有三个：

1. 文中补全要尽量像“插入式续写”，而不是泛泛的句子生成。
2. 文末补全要尽量快，不能明显拖慢正常写作节奏。
3. 用户明确触发补全时，系统尽量不要返回空结果。

---

## 2. 系统总览

当前文本补全是一条标准的前后端分离链路：

```text
用户输入 / 按快捷键
    ->
TiptapEditor 采集 prefix / suffix
    ->
POST /api/v1/completion
    ->
completion.py 校验参数并转发
    ->
llm_service.py 构造上下文、prompt、stop、重试策略
    ->
Ollama OpenAI 兼容接口 /v1/completions
    ->
返回补全文本
    ->
前端渲染 ghost text
    ->
用户 Tab 接受 / Esc 拒绝 / 继续输入取消
```

这里要强调一点：当前系统不是“全文续写器”，而是“光标位置补全器”。

也就是说，模型不是只看前文，而是要尽量结合：

- 光标前内容 `prefix`
- 光标后内容 `suffix`
- 当前触发方式 `manual` 或 `auto`

做出一段适合插入光标处的文本。

---

## 3. 两种补全模式

当前补全分成两种模式：

### 3.1 文中手动补全 `manual`

适用场景：

- 光标在文章中间
- 用户希望模型在前后文之间补一段桥接文本
- 用户主动按快捷键触发

设计目标：

- 质量优先
- 可以接受略高一点延迟
- 尽量利用更多上下文
- 尽量减少空补全

### 3.2 文末自动补全 `auto`

适用场景：

- 光标位于文章末尾
- 用户停止输入后，希望系统自动给出一句短续写

设计目标：

- 速度优先
- 结果要短
- 不能明显打断输入体验
- 不追求大段生成

这两种模式虽然都叫“文本补全”，但本质上是两套不同优化目标，所以当前实现把它们拆成了两套参数配置，而不是强行共用一套策略。

## 4. 前端实现

前端核心文件是：

- `frontend/src/components/Editor/TiptapEditor.vue`

前端主要负责四件事：

1. 判断什么时候触发补全
2. 采集当前光标前后的上下文
3. 管理 ghost text 生命周期
4. 在用户继续输入时及时取消旧补全请求

### 4.1 Ghost Text 状态

前端维护以下关键状态：

- `ghostText`
  当前显示的幽灵文本内容

- `ghostPos`
  幽灵文本插入的光标位置

- `completionTimer`
  文末自动补全的延迟触发定时器

- `completionRequestVersion`
  用于作废旧请求的版本号

- `activeCompletionAbortController`
  当前正在飞行中的请求控制器，用于主动取消

- `isApplyingExternalContent`
  用于区分“用户自己输入导致的内容变化”和“外部 setContent 导致的内容变化”

这套状态的核心思想是：

- 用户一继续输入，旧建议就失效
- 用户一移动光标，旧建议就失效
- 旧请求回来得再晚，也不能覆盖当前更近一次的上下文

### 4.2 触发方式

当前支持两类触发：

#### 手动触发

支持以下快捷键：

- `Ctrl/Cmd + Alt`
- `Ctrl/Cmd + Space`
- `Ctrl/Cmd + Enter`

其中：

- `Ctrl/Cmd + Space` 在某些系统或输入法环境里容易被系统层抢走
- `Ctrl/Cmd + Enter` 是本轮改动里额外加的兜底键，目的是降低“明确按了但没触发”的概率

#### 自动触发

自动触发只在“文章末尾”生效，且用户停止输入后延迟 `120ms` 触发。

当前判定“在末尾”的方式比较直接：

- 从当前光标开始往后看一小段文本
- 如果这段文本去掉空白后为空，则视为文末

这样设计的原因是：

- 不需要依赖特殊节点结构
- 对 Markdown/Tiptap 当前文档结构足够稳
- 判定成本低

### 4.3 为什么不让文中自动补全

这是刻意设计，不是能力缺失。

文中自动补全的问题在于：

- 用户只是切文档查看内容，也可能误触发
- 用户正在改局部表达，补全会频繁闪出来打断阅读
- 文中补全对语义连续性的要求更高，误触发成本更大

所以当前策略是：

- 文中补全必须手动触发
- 文末补全可以自动触发

### 4.4 上下文采集策略

前端采集上下文时已经按模式分流：

#### 手动补全

- `prefixWindow = 1280`
- `suffixWindow = 260`

#### 自动补全

- `prefixWindow = 640`
- `suffixWindow = 128`

这样做的目的很明确：

- 手动补全更像“认真补一段中间句”，需要看更多前后文
- 自动补全更像“快速补一句后续”，需要更轻量

### 4.5 为什么前端还要主动取消请求

因为补全接口天然存在延迟，而用户输入是连续的。

如果不取消旧请求，会出现典型问题：

1. 用户输入 A
2. 发出补全请求 A
3. 用户继续输入到 B
4. 发出补全请求 B
5. 请求 A 后回来，把旧结果插进当前上下文

这会直接造成：

- 建议错位
- 建议内容“慢半拍”
- 用户感知为补全不稳定

所以当前做法是双保险：

- 发新请求前，先 `abort` 当前请求
- 再通过 `completionRequestVersion` 版本号防止旧响应写回

## 5. API 层实现

后端接口文件是：

- `backend/app/api/completion.py`

接口输入结构目前包含：

- `prefix`
- `suffix`
- `language`
- `trigger_mode`

其中 `trigger_mode` 是本轮改动新增的关键参数。

它的作用不是给前端做展示，而是让后端明确知道：

- 这次是用户主动要求补文
- 还是文末自动给一个短建议

如果没有这个字段，后端只能用一套折中参数处理所有请求，质量和延迟都会被迫折中。

## 6. 后端补全服务实现

后端核心文件是：

- `backend/app/services/llm_service.py`

这部分承担真正的“补全质量控制”工作。

后端当前分成五层逻辑：

1. 模式归一化
2. 上下文裁剪
3. Prompt 构造
4. 模型调用
5. 后处理与质量重试

### 6.1 模式配置

当前定义了两套 `COMPLETION_PROFILES`。

#### `auto`

- `prefix_window = 640`
- `suffix_window = 128`
- `prompt_prefix_window = 360`
- `prompt_suffix_window = 96`
- `max_tokens = 24`
- `timeout = 7.0`
- `max_chars = 28`

特点：

- 轻量
- 快
- 返回更短

#### `manual`

- `prefix_window = 1200`
- `suffix_window = 240`
- `prompt_prefix_window = 720`
- `prompt_suffix_window = 160`
- `max_tokens = 40`
- `timeout = 8.5`
- `max_chars = 80`

特点：

- 上下文更长
- prompt 更完整
- 容忍更高一点模型耗时
- 允许输出稍长一点的桥接文本

这里要注意：

前端采集窗口和后端裁剪窗口不是完全一样的。

这是刻意设计：

- 前端负责尽量多带一些上下文
- 后端负责最后统一裁剪到真正要发给模型的规模

这样做的好处是：

- 前端逻辑简单
- 后端可以独立调参
- 不需要每次调 prompt 都同步改前端采样尺度

### 6.2 Prompt 策略

当前 prompt 有几个关键设计点。

#### 1. system prompt 明确限定角色

不是让模型“自由写作”，而是明确告诉模型：

- 你是中文写作补全器
- 只输出插入光标处的内容
- 不要解释
- 不要重复上下文
- 避免空泛套话

#### 2. user prompt 带前文、后文和明确要求

如果有 `suffix`，说明是中间补全，会明确告诉模型：

- 前文是什么
- 后文是什么
- 补文要夹在它们之间

如果没有 `suffix`，说明是文末续写，会告诉模型：

- 直接往下写
- 不要重新开题
- 不要空泛开头

#### 3. 增加 few-shot 示例

本轮改动里补充了小示例，目的是让模型更稳定地理解：

- “补全文本”不是解释任务
- “中间补全”应该像桥接语句
- “文末补全”应该像贴着前文往下写

#### 4. 重试时增加 retry hint

如果第一次结果为空、重复或太泛，第二次会在 prompt 里显式告诉模型：

- 上一次结果不合格
- 这次必须更具体
- 不要输出套话

这本质上是一个轻量的 self-correction。

### 6.3 Stop Sequences 策略

当前 stop 逻辑不是完全固定的。

系统会在通用 stop 之外，动态截取一段 `suffix` 作为附加 stop。

目的：

- 降低模型直接把后文复述出来的概率
- 降低中间补全时的 suffix echo

但这个 stop 只在第一次尝试里启用得更激进。

如果第一次失败，第二次重试会放松这部分约束，避免“因为 stop 太早导致结果总是被截断为空”。

这也是本轮改动里非常关键的一点。

---

## 7. 后处理与质量控制

补全系统真正难的地方，不在“调通模型”，而在“把模型输出变成可以直接给用户看的建议”。

当前后处理主要做以下几件事。

### 7.1 去前缀重叠

如果模型输出开头和前文末尾重复，会把重叠部分剪掉。

例子：

- 前文末尾已经是“提高财政收入”
- 模型又输出“提高财政收入和资源调配效率”
- 最终应只保留新增部分

### 7.2 去后缀重叠

如果模型输出尾部已经接近后文开头，也要剪掉。

这是中间补全最重要的稳定性逻辑之一。

### 7.3 suffix echo 检测

如果模型本质上只是把后文原样吐了一遍，当前实现会直接判定为无效。

这也是用户看到“按了快捷键却是空”的常见原因之一：

- 模型确实输出了东西
- 但那东西几乎等于后文
- 后处理为了避免错建议，把它清掉了

### 7.4 输出裁剪

自动补全和手动补全的裁剪方式不同：

- 自动补全更短，通常切到第一句或第一个明显停顿
- 手动补全允许稍长，但仍限制最大字符数，避免一口气生成整段

### 7.5 套话黑名单

当前实现显式把以下模式判为低质量：

- `在当今社会`
- `促进社会经济的发展`
- `推动社会经济的发展`
- `具有重要意义`
- `起到重要作用`
- `今天`
- `进一步`
- `值得注意的是`

这不是一个完美方案，但对当前中文写作场景很有效。

原因很简单：

- 这些短语过于常见
- 很多时候并不来自上下文，而是模型的默认写作惯性
- 一旦出现在补全建议里，用户会明显感觉“很 AI”

### 7.6 自动重试

如果第一次结果满足以下任一条件，系统会自动重试一次：

- 空结果
- 明显重复
- 明显套话
- 太短

注意：

这里只重试一次，没有无限重试。

原因是：

- 多次重试会显著拉长延迟
- 补全是编辑器里的高频交互，不能把它做成长推理任务

当前方案是在质量和时延之间做的折中。

## 8. 为什么用户还可能看到空补全

即使做了重试，当前系统仍然可能出现空补全，这在技术上是正常现象，不代表接口一定没触发。

主要原因有：

### 8.1 快捷键被系统或输入法拦截

尤其是：

- `Ctrl/Cmd + Space`

这个问题不是应用层代码能完全控制的。

所以当前才额外保留了：

- `Ctrl/Cmd + Enter`

作为更稳的兜底手动触发键。

### 8.2 模型输出被判为低质量

例如：

- 只是重复后文
- 输出套话
- 输出非常短且无信息量

这时系统宁愿返回空，也不返回一条明显糟糕的建议。

### 8.3 当前上下文信息不足

即使上下文窗口已经扩大，某些补全场景仍可能语义过弱，例如：

- 前文过短
- 光标附近缺少关键实体
- 后文本身很抽象

这类场景下模型即使有更长窗口，也未必能补出高质量文本。

### 8.4 当前模型本身的局限

当前补全模型本质上仍是本地通用模型，不是为中文编辑器 FIM 专门训练的定制模型。

所以它的典型弱点包括：

- 容易回到常见套话
- 对中间补全的“桥接性”理解不够稳定
- 对长句内部语气、逻辑、术语连续性的把握不总是稳

这不是纯靠 prompt 就能完全消除的问题。

## 9. 这轮改动的核心收益

从技术上看，这一轮改动主要解决了五类问题：

### 9.1 手动补全和自动补全不再混用同一套参数

这是最重要的架构变化。

如果两种模式继续共用一套策略，就会一直卡在：

- 想提速时，手动补全质量会掉
- 想提质时，自动补全会太慢

当前拆开后，至少参数层面已经可以独立优化。

### 9.2 前端补全状态机更稳定

当前状态机重点保证：

- 旧请求可取消
- 旧响应不可回写
- 切文档不会误触发
- 文末补全可以连续重新触发

### 9.3 手动补全上下文显著增加

这直接改善了“补全内容很一般”的问题。

因为文中补全最怕的是模型只看到一小截句子，然后按语言惯性胡补。

### 9.4 后端质量控制不再只靠单次生成

以前如果第一次生成很差，就直接空。

现在会：

- 先过滤
- 再重试
- 再返回最佳候选

这能明显降低“明确触发但什么都没有”的频率。

### 9.5 对中文套话做了显式抑制

这能明显改善“像 AI”但不贴文的感觉。

## 10. 真实调优结论

基于今天的调试结果，可以总结出几条经验。

### 10.1 中间补全不能只追求快

文中补全本质上是高约束生成。

如果上下文窗口太短、tokens 太少、prompt 太轻，模型大概率会：

- 输出泛词
- 输出桥接失败
- 输出后文重复

所以文中补全必须允许：

- 更长上下文
- 更长输出
- 更强约束
- 一次重试

### 10.2 文末补全必须快，但也不能短到只剩废话

文末补全如果太激进地裁剪，会出现：

- `今天`
- `进一步`
- `在未来`

这种几乎不能用的短提示。

所以当前自动补全已经比最早版本略放宽了：

- 上下文更长
- `max_tokens` 更大
- 输出允许稍长一点，但仍限制在短句级别

### 10.3 动态 stop 很有用，但不能过强

如果 stop 太弱，模型容易重复后文。

如果 stop 太强，模型可能刚起头就被截断，导致空结果。

所以当前实现里：

- 第一次尝试：更积极地利用 suffix 动态 stop
- 第二次尝试：放松这层限制

这是比较合理的折中。

## 11. 现阶段仍然存在的限制

当前系统已经比之前稳定很多，但还远没到“高质量中文补全产品级”。

主要限制如下。

### 11.1 仍依赖通用本地模型

当前没有使用专门做中文编辑器补全训练的模型，也没有做任务蒸馏或微调。

这意味着：

- 质量天花板有限
- prompt 能提升，但提升有限

### 11.2 仍缺少基于编辑行为的数据反馈

当前系统不知道：

- 用户接受了哪些补全
- 用户拒绝了哪些补全
- 哪类补全最容易被立即覆盖

如果未来要做真正可持续优化，这部分必须补。

### 11.3 仍缺少领域感知

当前补全没有真正区分：

- 议论文
- 学术写作
- 产品文案
- 技术说明

不同文体对补全风格要求非常不一样。

### 11.4 当前质量判断仍偏规则化

现在的低质量过滤主要依赖：

- 规则
- 长度阈值
- 套话列表
- 重复检测

这能挡住明显坏结果，但不等于真正理解“好不好”。

## 12. 后续建议

如果后续继续做补全，建议按以下优先级推进。

### 第一优先级：记录用户接受率

增加埋点：

- 触发次数
- 返回为空次数
- 用户接受次数
- 用户拒绝后继续输入的次数

没有这些数据，后面调参基本还是靠体感。

### 第二优先级：为 manual / auto 分别调 prompt

当前虽然已经拆开了，但还只是第一版。

后续可以继续细化：

- manual 更重视桥接性、术语延续、逻辑连贯
- auto 更重视自然下文、短句质量、节奏感

### 第三优先级：增加候选重排

现在是一条结果不行就重试一次。

未来可以考虑：

- 一次生成多个候选
- 用规则或轻量打分选择最优候选

这通常会比单次生成 + 强规则更稳。

### 第四优先级：引入文体提示

例如让前端在请求里附加：

- 当前文档标题
- 当前段落类型
- 用户是否处于列表/标题/正文区域

这些额外信号会显著帮助模型理解“该写什么风格”。

### 第五优先级：评估更适合中文补全的模型

如果后续质量仍不满意，最终还是模型问题。

这时最值得评估的是：

- 更擅长中文写作的本地模型
- 更适合 FIM/插入式补全的模型
- 对当前写作场景做轻量微调

---

## 13. 给后续维护者的结论

一句话总结当前补全系统：

> 这是一个“前端强状态控制 + 后端分模式 prompt + 规则过滤 + 一次自动重试”的中文编辑器补全系统。

它当前的优势是：

- 结构已经清楚
- manual / auto 已经解耦
- 触发链路比之前稳定
- 对空结果和套话有了基本控制

它当前的短板是：

- 质量仍然受限于基础模型
- 规则过滤仍偏硬编码
- 还没有数据反馈闭环

所以后续维护时，最重要的思路不是继续零碎修 bug，而是始终把补全看成三层系统：

1. 前端触发是否稳定
2. 后端上下文与 prompt 是否合理
3. 模型输出质量控制是否足够

如果只改其中一层，另外两层不动，最终体验很容易再次掉回“不稳定”或“很 AI”。

## 14. 当前关键参数速查表

### 前端

- 文末自动补全延迟：`120ms`
- 手动补全上下文：`prefix 1280 / suffix 260`
- 自动补全上下文：`prefix 640 / suffix 128`
- 请求超时：`9000ms`

### 后端 `manual`

- 裁剪窗口：`prefix 1200 / suffix 240`
- prompt 窗口：`prefix 720 / suffix 160`
- `max_tokens = 40`
- `timeout = 8.5`
- `max_chars = 80`
- 失败后自动重试 `1` 次

### 后端 `auto`

- 裁剪窗口：`prefix 640 / suffix 128`
- prompt 窗口：`prefix 360 / suffix 96`
- `max_tokens = 24`
- `timeout = 7.0`
- `max_chars = 28`
- 失败后自动重试 `1` 次

## 15. 最后说明

如果后续有人继续接手这条补全链路，建议优先记住下面三句话：

1. 文中补全和文末补全不是同一个问题，不能再混成一套参数。
2. 空补全很多时候不是没触发，而是质量控制故意把坏结果清掉了。
3. 现在这套实现已经能用，但离真正“写起来顺手”的中文补全，还有模型和数据层面的工作要做。
