# 消融实验说明

## 1. 实验目的

这次消融实验回答的是一个更细的问题：

1. 如果拿掉查询自适应权重，性能会掉多少
2. 如果拿掉覆盖率奖励，性能会掉多少
3. 如果拿掉标识符约束，性能会掉多少
4. 如果拿掉混合候选池，性能会掉多少

## 2. 为什么重做消融协议

前一版消融把 `RRF + 候选池30` 作为累加式对照，结果发现：

1. 单独扩大候选池本身就可能带来很强收益
2. 容易把“候选池大小变化”和“模块本身作用”混在一起

所以这一版改成更稳的“去模块消融”：

1. 以完整方案为中心
2. 每次只拿掉一个模块
3. 其他设置尽量保持不变

这样更适合做模块归因。

## 5. 结果解读

完整方案相对主流方案 B 的绝对提升为：

- `Hit@1`：`0.0150`
- `nDCG@10`：`0.0242`

相对“去掉自适应权重”版本，完整方案的提升为：

- `Hit@1`：`0.0167`
- `nDCG@10`：`0.0300`

相对“去掉覆盖率奖励”版本，完整方案的提升为：

- `Hit@1`：`0.0017`
- `nDCG@10`：`0.0038`

相对“去掉标识符约束”版本，完整方案的提升为：

- `Hit@1`：`-0.0008`
- `nDCG@10`：`-0.0017`

相对“去掉混合候选池”版本，完整方案的提升为：

- `Hit@1`：`0.0217`
- `nDCG@10`：`0.0535`

## 6. 当前最稳的结论口径

这轮消融最适合得出的结论是：

1. 查询自适应权重是主要贡献源之一。
2. 混合候选池是当前公开数据上最强的贡献项。
3. 覆盖率奖励有小幅正贡献，更像排序细化项。
4. 标识符约束在当前 `DuRetrieval` 上没有显示稳定正收益，不能硬讲成核心贡献。

因此，对外最稳的表达不是：

- “我们所有模块都有效”

而是：

- “完整方案的主要收益来自查询自适应权重和混合候选池，覆盖率奖励提供补充收益，标识符约束目前仍是待进一步验证的工程假设”
