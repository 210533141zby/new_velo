# 真实执行与排障日志

## 1. 这份文档的定位

这份文档专门记录这次 RAG benchmark 在真实执行过程中发生了什么。

它和其他几份文档的分工是：

- [详细流程与方法说明.md](/root/Velo/experiments/主流RAG检索对比实验/详细流程与方法说明.md)
  重点讲“方法、流程、指标、来源”。
- [方法改进与结果解读.md](/root/Velo/experiments/主流RAG检索对比实验/方法改进与结果解读.md)
  重点讲“方法到底改了什么、结果提升了多少、应该怎么解读”。
- [真实执行与排障日志.md](/root/Velo/experiments/主流RAG检索对比实验/真实执行与排障日志.md)
  重点讲“实验是怎么一步步跑起来的，中途出了什么问题，我怎么判断和处理”。

这份文档故意不写成“光滑的总结稿”，而是保留真实执行中的观察、判断和动作。

## 2. 记录格式

每条记录尽量统一成三段：

1. 观察到什么
2. 我怎么判断
3. 我做了什么动作

这样做的目的是让你以后答辩时，不只是会说“我做了 hybrid + rerank”，还能说清楚：

1. 我怎么判断实验在正常推进
2. 我怎么确认缓存真的复用了
3. 我怎么区分算法故障和服务故障
4. 我怎么把一次真实故障转化为脚本改进

## 3. 执行时间线总览

### 3.1 重做实验方向

观察：

1. 旧实验偏合成、偏弱基线。
2. 用“传统 chunk + rerank”做主对照，很容易出现夸张提升。

判断：

1. 这种结果不适合拿去复试简历。
2. 应该改成“主流 RAG 方案”对比，而不是继续打弱对手。

动作：

1. 删除旧的弱基线实验目录和旧脚本。
2. 保留 `experiments/` 根目录，重新组织实验材料。
3. 新建主脚本 [mainstream_rag_benchmark.py](/root/Velo/experiments/mainstream_rag_benchmark.py)。

### 3.2 换成公开中文 benchmark

观察：

1. 自造或小规模合成数据不够有说服力。
2. 用户明确希望能用于 985 复试，不希望结论显得夸张。

判断：

1. 应该改用公开中文 retrieval benchmark。
2. 数据必须可复现、可引用、规模足够。

动作：

1. 选定 `C-MTEB/DuRetrieval` 和 `C-MTEB/T2Retrieval`。
2. 脚本加入 parquet 下载和本地缓存逻辑。
3. 实际读取 parquet，确认字段结构与数据规模。

## 4. 数据核查阶段

### 4.1 先确认数据到底长什么样

观察：

我先直接读取缓存下来的 parquet，确认了：

1. `corpus` 只有 `id` 和 `text`
2. `queries` 只有 `id` 和 `text`
3. `qrels` 有 `qid`、`pid`、`score`

并且看到了实际示例文本，确认它们确实是中文检索数据，不是空壳。

判断：

1. 数据结构干净，适合直接做 retrieval benchmark。
2. 需要做文本清洗，因为原文里有 HTML 标记、`<br>`、图片残留等网页噪声。

动作：

1. 脚本里保留 `clean_text()`。
2. 脚本里保留 `tokenize_text()`，为 BM25 做分词准备。
3. 文档文本在进入 embedding 前截到合理长度，避免超长文本拖垮服务。

### 4.2 规模确认

观察：

脚本实际看到的规模是：

1. `DuRetrieval` 约 `100001` 篇文档、`2000` 条查询、`9839` 条 qrels
2. `T2Retrieval` 约 `118605` 篇文档、`22812` 条查询、`118932` 条 qrels

判断：

1. 这已经足够构成正式 benchmark，不再是“小打小闹”的数据量。
2. 全量直接跑会太慢，所以需要固定抽样查询数。

动作：

1. 每个数据集默认采样 `1200` 条查询。
2. 固定随机种子 `20260415`。

## 5. 模型与服务选择阶段

### 5.1 embedding 模型怎么选

观察：

本机有：

1. `bge-m3:latest`
2. `nomic-embed-text:latest`
3. `kopens/bge-reranker-v2-m3:latest`

其中 `bge-m3` 更像中文公开 benchmark 里更自然的选择，但通过 Ollama embedding API 的大规模稳定性不理想。

判断：

1. 正式 benchmark 的核心是“整条检索链路比较”，不是单模型极限刷分。
2. 如果 embedding 服务不稳定，整轮 benchmark 就会频繁中断，反而拖垮正式结果。

动作：

1. 先保留 `BAAI/bge-reranker-v2-m3` 作为 reranker。
2. 正式 embedding 暂时选 `nomic-embed-text:latest`，优先保证稳定可复现。
3. 脚本支持通过 `--embedding-model` 显式指定模型，后续可替换。

### 5.2 为什么不直接把 `bge-m3` 写死

观察：

之前尝试 `bge-m3:latest` 时，Ollama API 在正式长跑里出现过 `HTTP 500`。

判断：

1. 它不是完全不能用，但至少在当前机器和当前服务链路上不稳定。
2. 正式实验阶段不适合继续赌运气。

动作：

1. 选择 `nomic-embed-text:latest` 做正式跑数。
2. 保留 `bge-m3` 作为后续增强对比或替换候选，而不是现在的阻塞项。

## 6. smoke 测试阶段

判断：

1. 需要先用小语料、小查询做一轮端到端测试。
2. 重点不是看指标有多高，而是看流程是否完整、输出是否齐全。

动作：

1. 运行 `DuRetrieval` 的小规模 smoke。
2. 参数大致是 `query-limit=20`、`corpus-limit=4000`。
3. 检查生成的 CSV、JSON、SVG、Notebook 和方法结果总结文档。

### 6.2 smoke 阶段得到了什么

观察：

smoke 运行成功，输出了：

1. `metrics_by_dataset.csv`
2. `per_query_results.json`
3. `representative_queries.csv`
4. `overall_metrics.svg`
5. `latency_quality.svg`
6. `主流RAG方案对比实验.ipynb`

判断：

1. 主脚本整体链路是通的。
2. 新的“主流 RAG 方案”命名和输出目录结构都已经落地。

动作：

2. 正式实验继续用更大的公开 benchmark 跑数。

## 7. 第一次正式运行

### 7.1 先跑 `DuRetrieval`

观察：

正式 run 启动后，`DuRetrieval` 的文档规模是 `100001`，这是第一段最耗时部分。

判断：

1. 首轮最重的不是 rerank，而是文档 embedding 建库。
2. 只要这一段完成，后续重跑速度会快很多。

动作：

1. 先盯 `DuRetrieval` doc embedding 是否开始生成缓存。
2. 观察缓存目录变化。

### 7.2 为什么中途选择“暂停”

观察：

第一次正式运行中，`DuRetrieval` 的文档和查询 embedding 已经落盘，逐查询评测推进到大约 `600 / 1200`。

判断：

1. 这时候暂停是安全的，因为最重的 embedding 缓存已经保住。
2. 继续硬跑意义不大，用户已经明确表示可以明天再继续。

动作：

1. 中断进程。
2. 当时单独写过一份暂停恢复备注，用来记录缓存和恢复命令。
3. 明确记录缓存文件路径和恢复命令。

## 8. 恢复实验阶段

### 8.1 恢复前不直接跑，而是先核对

观察：

第二天恢复前，存在两个风险：

1. 可能缓存没了
2. 可能有旧 benchmark 进程残留

判断：

1. 不先核对，恢复实验就不可信。
2. 如果旧进程和新进程同时写同一个输出目录，结果会被污染。

动作：

1. 先读恢复说明。
2. 检查 `DuRetrieval` 的 doc/query embedding 缓存。
3. 查后台残留 benchmark 进程。

### 8.2 为什么特别关注“缓存命中”

观察：

恢复后日志里出现了：

1. `reuse cached doc embeddings`
2. `reuse cached query embeddings`

判断：

1. 说明昨天最重的向量化工作没有白做。
2. 今天能直接进入 `DuRetrieval` 的检索评测阶段。

动作：

1. 继续盯逐查询进度。
2. 把注意力放到 rerank 性能和第二套数据集切换上。

## 9. `DuRetrieval` 正式评测阶段

### 9.1 为什么要不断记录 `100/1200` 这种进度

观察：

日志依次出现：

1. `processed 100/1200`
2. `processed 200/1200`
3. `processed 300/1200`
4. 一直到 `1100/1200`

判断：

这些进度不只是“报数”，还用来判断：

1. 程序是不是还活着
2. 速度是不是突然异常变慢
3. 是否已经越过昨天的中断点
4. 是否快进入下一阶段风险点

动作：

1. 持续监控进度。
2. 在超过 `600/1200` 后确认今天开始产生了新的净进度。
3. 在接近 `1100/1200` 时，提前转向关注 `T2Retrieval`。

### 9.2 为什么我会去看 GPU 利用率

观察：

有些阶段日志不会立刻刷新。

判断：

1. 如果 GPU 接近满载，说明 dense 排序或 rerank 很可能在正常推进。
2. 如果 GPU 空闲、没有新缓存、没有新日志，就要怀疑是不是程序停在别处或挂掉。

动作：

1. 用 `nvidia-smi` 看 GPU 利用率和显存占用。
2. 结合进程和缓存目录一起判断“是慢”还是“是挂”。

## 10. `DuRetrieval` 完成后的切换

### 10.1 为什么 `DuRetrieval` 跑完后不算结束

观察：

`DuRetrieval` 完成以后，程序切到 `T2Retrieval`。

判断：

1. 第一套数据集完成，只说明主链路基本没问题。
2. 第二套数据集才是新的不确定阶段，因为它没有现成 embedding 缓存。

动作：

1. 开始关注 `t2` 缓存是否开始生成。
2. 关注 GPU 利用率是否重新升高。

### 10.2 为什么“暂时没新日志”不一定是坏事

观察：

切到 `T2Retrieval` 后，短时间内没有新日志。

判断：

1. 有可能是在做 CPU 侧分词、BM25 构建和前置准备。
2. 不能因为一分钟没日志就马上宣布失败。

动作：

1. 查看 `t2_*` 缓存文件有没有开始生成。
2. 查看 GPU 是否重新工作。

## 11. 第一次显式故障

### 11.1 故障现象

观察：

程序在 `T2Retrieval` 开始做 doc embedding 时报错：

- `Connection refused`
- 地址是 `127.0.0.1:11434`

判断：

这不是：

1. BM25 公式错了
2. rerank 模型错了
3. 数据集损坏了

而是：

1. 本地 embedding 服务不可达
2. 更具体地说，是 `ollama serve` 掉了

### 11.2 为什么我能判断这是服务问题，不是算法问题

观察：

报错位置发生在 HTTP embedding 请求阶段，而且错误类型是连接拒绝。

判断：

1. 如果是算法公式、排序逻辑或数据格式问题，错误一般不会表现成 API 端口连接拒绝。
2. 连接拒绝更像是服务没启动、服务退出或端口没监听。

动作：

1. 查 `ollama` 进程。
2. 用 `curl` 请求 `http://127.0.0.1:11434/api/tags`。
3. 确认确实连不上。

## 12. 服务恢复阶段

### 12.1 恢复服务前先确认有没有脏缓存

观察：

在 `t2` 故障后，缓存目录里还没有新的 `t2` embedding 文件。

判断：

1. 这说明还没产生半成品缓存。
2. 可以直接恢复服务后重新开始，不需要先清理脏状态。

动作：

1. 检查 `t2_*` 缓存是否存在。
2. 确认只有 parquet，没有新的 `.npy` embedding 文件。

### 12.2 恢复服务

观察：

`curl` 访问 embedding API 失败，说明服务确实不在线。

判断：

1. 需要先恢复 `ollama serve`，否则继续跑脚本没有意义。

动作：

1. 重新启动 `ollama serve`
2. 再次请求 `api/tags`
3. 确认模型列表正常返回

## 13. 为什么我又顺手改脚本

### 13.1 问题出在哪里

观察：

虽然 `DuRetrieval` 已经完整跑完，但旧版脚本是整轮都结束后才统一落盘。

判断：

1. 只要 `T2Retrieval` 出故障，`DuRetrieval` 的正式结果也会跟着丢。
2. 这不利于正式 benchmark，也不利于工程复现。

动作：

1. 把输出写盘逻辑抽成 `write_benchmark_outputs()`
2. 每完成一个数据集，就先做一次阶段性落盘
3. 这样后半程再挂，前半程结果也能保住

### 13.2 这不是“理论升级”，但很重要

观察：

这个改动不会直接提升 `Hit@1` 或 `nDCG@10`。

判断：

1. 它属于 benchmark 工程化增强，不属于检索算法创新。
2. 但它直接影响实验结果是否可靠、是否能恢复。

动作：

1. 把这类小改动也写进文档。
2. 不把所有工作都包装成“理论突破”。

## 18. 正式结果之后的真实收尾

### 18.1 为什么正式结果出来后还不能立刻收工

观察：

主实验虽然已经完整跑完，但还剩三个现实问题：

1. 还没有严谨回答“到底是哪几个模块在贡献收益”。
2. 图表和材料还没有完全适配中文展示和复试表达。
3. 算法还没有真正接到 backend 主链路里。

判断：

1. 只有主结果，没有消融，老师很容易追问“是不是堆模块堆出来的”。
2. 只有离线 benchmark，没有系统落地，也容易被追问“是不是只是脚本实验”。

动作：

1. 单独补了一轮去模块消融实验。
2. 修了一轮实验产物和文档口径。
3. 最后再把算法接到 backend 实际 RAG。

### 18.2 为什么把第一版消融协议推翻重做

观察：

最初那版消融更像“累加式”设计，把 `RRF + 候选池 30` 这样的设置堆在一起看。

判断：

1. 单独扩大候选池本身就可能显著影响结果。
2. 这样会把“模块作用”和“候选池大小变化”混在一起，模块归因不干净。

动作：

1. 放弃第一版口径，不直接拿它写结论。
2. 重构 [ablation_study.py](/root/Velo/experiments/ablation_study.py)，改成“以完整方案为中心、每次只去掉一个模块”的去模块消融协议。
3. 把 `主流方案 B` 保留成外部参考线，而不是让它和模块消融混为一谈。

### 18.3 去模块消融跑完后，真正得到的是什么

观察：

消融跑完后，`DuRetrieval` 上的结果很明确：

1. 去掉查询自适应权重，指标明显下降。
2. 去掉混合候选池，指标下降最大。
3. 去掉覆盖率奖励，小幅下降。
4. 去掉标识符约束，均值没有变差，甚至略有变好。

判断：

1. 主要贡献源已经可以明确讲成“查询自适应权重”和“混合候选池”。
2. 覆盖率奖励可以讲成补充项。
3. 标识符约束不能再硬讲成稳定正贡献。

动作：

1. 把这个负结果保留下来，不为了好看重跑到“涨分”为止。
2. 在主文档里同步改口径，避免后面复试时前后说法不一致。

### 18.4 中途遇到的两个材料级故障

观察：

消融结果出来后，又遇到了两个并不属于算法本身的问题：

1. 生成说明表格时缺少 `tabulate` 依赖。
2. `ablation_core_metrics.svg` 中文显示成方框乱码。

判断：

1. 这两个问题不影响原始 CSV 和 JSON 结果。
2. 但会直接影响材料可读性和复试展示效果。

动作：

1. 去掉 `ablation_study.py` 里对 `tabulate` 的强依赖，避免因为一个展示库导致结果脚本失败。
2. 下载本地 CJK 字体 [SourceHanSansSC-Regular.otf](/root/Velo/experiments/.assets/fonts/SourceHanSansSC-Regular.otf)。
3. 把字体绑定进 SVG 生成逻辑，重新生成中文版本的消融图。

### 18.5 为什么我没有为了“标识符约束是负贡献”去重跑

观察：

用户看到消融结果后，第一反应很自然：

1. 标识符约束看起来像负作用，要不要改一版再跑，争取变成正的？

判断：

1. 如果只是为了让图好看而改实验口径，这个结果就不再可信。
2. 当前更稳的做法是承认：它是一个工程假设，但在这套公开 benchmark 上没有被稳定验证。

动作：

1. 不再为了“跑出好看正贡献”而反复调标识符约束。
2. 只在系统落地时，把它从硬惩罚收敛成轻量影响。

### 18.6 算法如何真正接到 backend

观察：

当时 backend 的实际 RAG 还不是实验里的标准自适应混合检索链路，更像：

1. Chroma 向量召回
2. 标题候选补充
3. rerank
4. 一些较重的 heuristic 最终打分

判断：

1. 如果停在这里，实验和系统会脱节。
2. 应该把实验里的“自适应 hybrid + rerank”真正落到 backend 主链路。

动作：

1. 新增 [hybrid_retrieval.py](/root/Velo/backend/app/services/ai/hybrid_retrieval.py)，实现文档级 BM25、查询画像、RRF、自适应融合和混合候选池。
2. 修改 [agent_service.py](/root/Velo/backend/app/services/ai/agent_service.py) 的 `rag_qa()`，把主流程改成 `向量召回 + BM25 + Adaptive Hybrid + Rerank`。
3. 修改 [rerank_service.py](/root/Velo/backend/app/services/ai/rerank_service.py)，把过去过重的标识符硬惩罚降成轻量调整。
4. 更新 [config.py](/root/Velo/backend/app/core/config.py)，启用更贴近实验设置的检索参数。
5. 补充 backend 单元测试并在项目虚拟环境下通过。
