## 3. 对比的三条检索链路

### 3.1 主流 RAG 方案 A

`Dense Retrieval + Rerank`

流程为：

1. 先做向量召回。
2. 取 dense `Top 20` 候选。
3. 用 cross-encoder reranker 精排。
4. 输出最终 `Top 10`。

这是最常见的单路语义检索链路。

### 3.2 主流 RAG 方案 B

`Hybrid Retrieval + RRF + Rerank`

流程为：

1. dense 召回 `Top 50`
2. BM25 召回 `Top 50`
3. 用 RRF 融合两路排序
4. 取 RRF 后 `Top 20` 候选
5. 用 reranker 精排
6. 输出最终 `Top 10`

这是更接近今天主流工业 RAG 检索的混合方案。

### 3.3 我们的方案

`Adaptive Hybrid + Multi-signal Fusion + Rerank`

在主流方案 B 基础上，我们额外做了四件事：

1. 查询自适应权重
2. 词法覆盖率奖励
3. 标识符一致性奖励与惩罚
4. 更稳的混合候选池构造

所以这次实验不是在“弱基线”上取巧，而是在已经较强的混合检索链路上继续优化。

## 4. 我们的方法到底多做了什么

这一节不只讲概念，而是给出脚本中的正式定义。

### 4.1 Dense 相似度

查询和文档都先被编码成 embedding，并做 L2 归一化。归一化后，dense 相似度直接使用点积：

```text
s_dense(q, d) = <q_hat, d_hat>
```

其中：

1. `q_hat` 是归一化后的查询向量。
2. `d_hat` 是归一化后的文档向量。
3. 因为都做了 L2 归一化，所以该点积等价于 cosine similarity。

### 4.2 BM25 词法相关性

BM25 用于建模精确词项匹配，尤其适合：

1. 专有名词
2. 编号和型号
3. 年份和数值
4. 中文关键短语

其标准形式为：

```text
BM25(q, d) = Σ IDF(t) * (f(t, d) * (k1 + 1)) / (f(t, d) + k1 * (1 - b + b * |d| / avgdl))
```

脚本中直接使用 `BM25Okapi` 默认实现，没有额外对 `k1` 和 `b` 做人工调参。

### 4.3 RRF 排名融合

对于 dense 排序与 BM25 排序，RRF 分数定义为：

```text
s_rrf(d) = Σ 1 / (k + rank_m(d))
```

其中：

1. `rank_m(d)` 是文档 `d` 在某个排序列表中的名次。
2. `k = 60` 是本实验固定使用的平滑常数。

RRF 的优点是：

1. 不要求 dense 分数和 BM25 分数量纲一致。
2. 对“在多个列表里都排得较靠前”的文档更友好。
3. 非常适合混合召回阶段的稳定融合。

### 4.4 查询自适应权重

主流 hybrid 常见做法是固定融合权重，但我们的脚本先计算查询画像，再动态设置 lexical 权重。

代码中的规则是：

```text
w_lex 初始为 0.36
若查询含编号或字母标识符，w_lex += 0.18
若查询分词数 <= 4，w_lex += 0.14
若查询分词数 >= 10，w_lex -= 0.08
若存在高 IDF 稀有词，w_lex += 0.10
最后将 w_lex 裁剪到 [0.24, 0.78]
w_dense = 1 - w_lex
```

也就是说：

1. 短查询更偏词法检索。
2. 带编号、型号、年份的问题更偏词法检索。
3. 更长、更抽象的问题再把比重往 dense 侧拉回去。

这一步主要针对的是排序前几名，因此通常更直接改善：

1. `Hit@1`
2. `MRR@10`

### 4.5 词法覆盖率奖励

对于查询分词集合 `T(q)` 和候选文档分词集合 `T(d)`，脚本定义：

```text
coverage(q, d) = |T(q) ∩ T(d)| / |T(q)|
```

这个量衡量的是：

查询中的核心词，到底有多少在文档里真正出现。

它主要用于修正下面这种错误：

1. 语义上看起来像，但关键问题词覆盖不够。
2. 文档主题接近，但并不是最精确的证据文档。

### 4.6 标识符一致性约束

对于查询中的编号、版本号、课程号、年份、型号等模式，脚本先抽取标识符集合 `I(q)` 和 `I(d)`，再计算：

```text
identifier_overlap(q, d) = |I(q) ∩ I(d)| / |I(q)|
```

如果查询本身包含标识符，但候选文档完全没有对应重合，则额外施加惩罚：

```text
penalty(q, d) = -0.10
```

这一项主要修正的是：

1. 只差一个数字就完全答错的问题
2. 型号近似但不一致的问题
3. 版本或年份错位的问题

### 4.7 最终融合分数

在候选集合上，脚本先对 dense、BM25、RRF 分数分别做 min-max 归一化：

```text
norm(x_i) = (x_i - min(x)) / (max(x) - min(x))
```

若某一路分数在候选中全部相同，则统一记为 `1.0`。

最终，自适应融合分数定义为：

```text
s_adaptive(q, d)
= w_dense * norm(s_dense)
+ w_lex * norm(s_bm25)
+ 0.26 * norm(s_rrf)
+ 0.08 * coverage(q, d)
+ 0.06 * identifier_overlap(q, d)
- 0.10 * 1[I(q) 非空 且 identifier_overlap(q, d) = 0]
```

这说明我们的方案不是训练型 learning-to-rank，而是：

1. 可解释
2. 可复现
3. 便于答辩说明
4. 工程实现成本低

### 4.8 更稳的混合候选池

主流方案 B 是直接把 RRF `Top 20` 丢给 reranker。

我们的方案不是这样。脚本中的候选池构造逻辑是：

1. 先取自适应融合排序前 `19` 个候选。
2. 再取 RRF 排序前 `19` 个候选补充。
3. 若还未达到上限，再继续按自适应排序补齐。
4. 最终送入 rerank 的候选上限为 `30`。

它的实际作用是：

1. 用更精细的粗排提升候选纯度。
2. 同时保留来自不同路径的高价值文档，避免过早误删。
3. 让 cross-encoder 在更干净、但不至于过窄的候选池上发挥作用。

### 4.9 为什么这一步属于启发式融合，而不是训练型排序

这一点一定要讲清楚，因为老师很可能会追问“你到底有没有训练模型”。

本实验中，真正属于我们方法创新的部分，是上面这套：

1. 查询画像规则
2. 多信号打分公式
3. 候选池构造策略

这些部分都没有在 `DuRetrieval` 或 `T2Retrieval` 的 qrels 上做参数学习，也没有做：

1. 梯度更新
2. 排序损失优化
3. 训练集 / 验证集 / 测试集划分后的监督拟合

也就是说：

1. qrels 只用于最终评测，不用于训练我们的方法参数。
2. `0.36 / 0.18 / 0.14 / 0.08 / 0.10 / 0.26 / 0.08 / 0.06 / -0.10` 这些系数是启发式设定，不是用监督学习拟合出来的。
3. 因此这部分本质上是“可解释的启发式融合”，而不是“任务特定的训练型 learning-to-rank”。

但也要讲得更准确一点：

1. 整条链路里的 reranker `BAAI/bge-reranker-v2-m3` 本身是一个公开的训练型 cross-encoder 模型。
2. 不过我们没有在当前 benchmark 上重新训练它。
3. 所以更准确的说法是：我们的改进部分属于“启发式检索融合”，而不是“基于当前数据集重新训练得到的学习排序器”。

## 5. 指标定义与统计方式

### 5.1 Hit@1

设查询 `q` 的最终排序结果为 `R_q`，相关文档集合为 `G_q`，则：

```text
Hit@1(q) = 1, 若 R_q[1] ∈ G_q
Hit@1(q) = 0, 其他情况
```

整体结果为所有查询的平均值。

### 5.2 MRR@10

设第一个相关文档在前 `10` 名中的位置为 `r_q`，则：

```text
MRR@10(q) = 1 / r_q, 若 r_q <= 10
MRR@10(q) = 0, 若前 10 名没有相关文档
```

整体结果为所有查询的平均值。

### 5.3 Recall@10

```text
Recall@10(q) = |Top10(q) ∩ G_q| / |G_q|
```

它衡量的是前 `10` 个结果覆盖了多少标准相关文档。

### 5.4 nDCG@10

脚本中使用二值相关性，定义为：

```text
DCG@10(q) = Σ 1 / log2(rank + 1)
```

其中求和只对前 `10` 名中的相关文档进行。

再定义：

```text
IDCG@10(q) = 前 min(|G_q|, 10) 个位置的理想 DCG
nDCG@10(q) = DCG@10(q) / IDCG@10(q)
```

这个指标同时考虑：

1. 是否找到了正确文档
2. 正确文档排得是否足够靠前

### 5.5 平均时延

对每个查询，脚本记录从候选构造到 rerank 完成的整条检索链路耗时，单位为毫秒，最后对查询求平均：

```text
AvgLatency = mean(latency_ms(q))
```

### 5.6 95% Bootstrap 置信区间

本实验对 `Hit@1` 和 `nDCG@10` 采用 `1000` 次 bootstrap 重采样估计均值的 95% 置信区间。

这一步的目的不是把结果“包装得更复杂”，而是为了给出：

1. 点估计
2. 波动范围
3. 更接近正式 benchmark 的统计表达

### 5.7 从单条查询到最终表格，分数是怎么汇总出来的

老师如果追问“你这个表到底怎么算出来的”，你就按这一节回答。

对于某个数据集 `D`、某条检索链路 `P` 和某个评测指标 `m`，脚本先对每个查询分别计算单条查询得分：

```text
m(q, P)
```

再对该数据集中的全部采样查询求平均，得到数据集级得分：

```text
Score(D, P, m) = (1 / |Q_D|) * Σ_{q ∈ Q_D} m(q, P)
```

其中：

1. `Q_D` 是数据集 `D` 中参与正式评测的查询集合。
2. 本实验里 `DuRetrieval` 和 `T2Retrieval` 都固定为 `1200` 条查询。

跨数据集的“平均表现”再继续做一次平均：

```text
Overall(P, m) = (1 / |Datasets|) * Σ_{D} Score(D, P, m)
```

因为这次两个数据集的查询数都同为 `1200`，所以这里的跨数据集平均，也等价于对两套数据的结果做等权汇总。

### 5.8 我们说“有提升”，到底是怎么计算出来的

本实验中的“提升”默认指相对主流方案 B 的绝对增量：

```text
Delta_abs(D, m) = Score(D, Ours, m) - Score(D, MainstreamB, m)
```

跨数据集平均提升为：

```text
Delta_abs(m) = Overall(Ours, m) - Overall(MainstreamB, m)
```

如果需要写成相对提升率，也可以再除以基线得分：

```text
Delta_rel(m) = Delta_abs(m) / Overall(MainstreamB, m) * 100%
```

但这次正式文档里，我们主报的是“绝对提升”，原因是：

1. 检索排序指标本身已经在 `[0, 1]` 区间内。
2. 用绝对提升更容易解释成“提升了多少个百分点”。
3. 对复试表达来说，绝对增量通常比相对百分比更稳。

例如，`DuRetrieval` 上的 `Hit@1`：

```text
0.8558 - 0.8408 = 0.0150
```

也就是：

- 绝对提升 `1.50` 个百分点

再例如，跨两套数据的平均 `nDCG@10`：

```text
0.6921 - 0.6760 = 0.0161
```

也就是：

- 绝对提升 `1.61` 个百分点

### 5.9 评测公平性是怎么控制的

为了避免“提升只是因为设置不一样”，这次实验在评测协议上做了严格控制。

所有 pipeline 共用：

1. 相同的数据集与相同语料库
2. 相同的采样查询集合
3. 相同的 qrels 标注
4. 相同的文本清洗和分词逻辑
5. 相同的 embedding 模型
6. 相同的 reranker
7. 相同的初始召回深度 `Top 50`
8. 相同的最终评测深度 `Top 10`

真正变化的只有：

1. 候选如何融合
2. 候选如何在 rerank 前筛选
3. rerank 之前送入的候选池大小

另外要注意：

1. embedding 缓存只影响运行速度，不影响最终分数。
2. qrels 只用于评测，不用于训练我们的融合参数。
3. 因此，这是一组受控条件下的公平对比，不是“换一套模型再拿结果硬比”。

这轮消融没有继续使用早期那种“累加式弱控制”设计，而是改成了更稳的协议：

1. 以完整方案为中心。
2. 每次只去掉一个模块。
3. 其他设置尽量保持不变。
4. 再保留 `主流方案 B` 作为外部参考线。

对某个模块 `x`，这里报告的模块贡献定义为：

```text
Delta_module(x, m) = Score(full, m) - Score(full without x, m)
```

并且对单条查询差值做 paired bootstrap，给出 95% 区间。更稳的解释原则是：

1. 如果区间明显高于 `0`，说明这个模块在当前数据上有稳定正贡献。
2. 如果区间跨过 `0`，说明当前证据不足，不能把它讲成“已经被公开 benchmark 稳定验证”。

| --- | --- | --- | --- | --- | --- |
| 主流方案 B（RRF + Rerank Top20） | 0.8408 | 0.6616 | +0.0150 | +0.0242 | 完整方案相对主流混合检索仍有稳定增益 |
| 去掉自适应权重（固定 lexical/dense 权重） | 0.8392 | 0.6558 | +0.0167 | +0.0300 | 查询自适应权重是主要贡献源之一 |
| 去掉覆盖率奖励 | 0.8542 | 0.6820 | +0.0017 | +0.0038 | 覆盖率奖励贡献较小，但 nDCG 上为正 |
| 去掉标识符约束 | 0.8567 | 0.6874 | -0.0008 | -0.0017 | 当前公开数据上未显示稳定正贡献 |
| 去掉混合候选池 | 0.8342 | 0.6323 | +0.0217 | +0.0535 | 混合候选池是最强贡献源之一 |
| 完整方案 | 0.8558 | 0.6858 | 0.0000 | 0.0000 | 最终完整版本 |

如果再结合 paired bootstrap 区间看，可以得到更严格的结论：

1. 去掉自适应权重后，`Hit@1` 和 `nDCG@10` 都明显下降，说明查询自适应权重贡献稳定。
2. 去掉混合候选池后，`Hit@1` 和 `nDCG@10` 下降最大，说明候选池设计是当前公开数据上最强的贡献项。
3. 去掉覆盖率奖励后，`nDCG@10` 仍有小幅稳定下降，但 `Hit@1` 区间接近 `0`，说明它更像是“排序细化项”，而不是主要驱动项。
4. 去掉标识符约束后，区间跨过 `0`，甚至均值略优于完整方案，说明这项设计目前还不能在 `DuRetrieval` 上被表述为稳定正收益。

### 6.6 消融实验应该怎么解读才诚实

这轮消融最适合得出的结论，不是“所有模块都有效”，而是：

1. 公开 benchmark 上，完整方案的主要增益来源是查询自适应权重和混合候选池。
2. 覆盖率奖励提供了小幅但方向正确的补充收益。
3. 标识符约束更像是面向企业知识库中编号、型号、版本号查询的工程假设；在当前 `DuRetrieval` 公开数据上，它还没有被稳定验证成通用正贡献。

这个结论比“所有模块都涨分”的故事更可信，也更像真实做过检索实验的人会给出的判断。

## 7. 指标提升应该怎么解释

这一部分不是空泛地说“效果更好了”，而是把“哪个模块更可能拉动哪个指标”讲清楚。

### 7.1 为什么 `Hit@1` 和 `MRR@10` 会提高

`Hit@1` 和 `MRR@10` 本质上都更关注“第一个正确文档来得早不早”。

因此它们最容易受下面两项影响：

1. 查询自适应权重
2. 更稳的混合候选池

原因是：

1. 查询自适应权重会在短查询、编号查询上，把 BM25 的作用放大，让正确文档更早进入前几名。
2. 混合候选池会减少粗排阶段的误删，让本来会被提前漏掉的正确文档还能进入 rerank。

从已经补完的 `DuRetrieval` 消融来看：

1. 去掉自适应权重后，`Hit@1` 下降 `1.67` 个百分点。
2. 去掉混合候选池后，`Hit@1` 下降 `2.17` 个百分点。
3. 标识符约束在当前公开数据上没有显示稳定正向贡献，因此不能把它硬讲成主要来源。

### 7.2 为什么 `Recall@10` 和 `nDCG@10` 也会提高

`Recall@10` 和 `nDCG@10` 不只看第 1 名，而是看前 `10` 个结果的整体质量。

因此它们更容易受下面两项影响：

1. 更稳的混合候选池
2. 查询自适应权重与覆盖率奖励的组合

原因是：

1. 更稳的候选池能给 reranker 更好的输入分布，减少早期筛选造成的漏召回。
2. 查询自适应权重和覆盖率奖励一起，会把“关键词更对、语义也更对”的候选往前稳住。

这一点在主实验和消融里都是对得上的：

1. DuRetrieval 上 `Recall@10` 提升 `+0.0264`
2. DuRetrieval 上 `nDCG@10` 提升 `+0.0242`
3. 去掉混合候选池后，DuRetrieval 的 `nDCG@10` 下降 `5.35` 个百分点，是所有模块里影响最大的
4. 去掉自适应权重后，DuRetrieval 的 `nDCG@10` 下降 `3.00` 个百分点
5. 去掉覆盖率奖励后，DuRetrieval 的 `nDCG@10` 下降 `0.38` 个百分点，说明它是补充项，不是主导项
6. T2Retrieval 上整体结果仍保持正提升，只是幅度更温和

### 7.3 为什么不是“夸张碾压”

这恰恰说明实验更可信。

如果我们拿弱基线或合成数据，很容易出现特别夸张的提升。但这次：

1. 对手是主流混合检索强基线
2. 数据是公开中文 benchmark
3. 提升在两套数据上都为正
4. 幅度稳定，但没有不合理夸大

这更像一个真实检索系统优化实验，而不是“故意挑一个好赢的对手”。

### 7.4 为什么时延会增加

时延增加主要来自两部分：

1. 自适应多信号打分本身比单纯 RRF 更复杂
2. 我们把 rerank 候选池从 `20` 扩到了 `30`

所以更稳的表达不是“几乎没有代价”，而是：

1. 我们在主流混合检索强基线之上，获得了稳定质量提升
2. 代价是平均约 `80.20 ms/query` 的额外时延
3. 这是一种可解释、可接受、可继续优化的工程 trade-off

## 8. 启发式融合和训练型 Learning-to-Rank 的区别

这一节是给你答辩时防追问用的。

### 8.1 启发式融合是什么

启发式融合的意思是：

1. 先人工设计特征
2. 再人工设定打分规则和权重
3. 不用监督学习去拟合参数

在我们这个实验里，启发式部分就是：

1. 查询自适应权重规则
2. 覆盖率奖励
3. 标识符一致性奖励与惩罚
4. 候选池混合策略

它的优点是：

1. 不需要额外标注训练集
2. 解释性很强
3. 改动成本低，适合做系统优化实验

它的不足是：

1. 权重依赖人工经验
2. 上限通常不如专门训练的排序模型高
3. 很难自动学出更复杂的非线性交互

### 8.2 训练型 Learning-to-Rank 是什么

训练型 learning-to-rank 的核心是：

1. 先构造带标签的训练样本，例如 `(query, doc, relevance)`
2. 再让模型根据这些标注去学习排序函数
3. 最终用学习到的函数替代人工规则

常见形式包括：

1. pointwise：直接回归或分类相关性分数
2. pairwise：学习“文档 A 应该排在文档 B 前面”
3. listwise：直接优化整个排序列表

这类方法常见的代表有：

1. LambdaMART
2. RankNet / LambdaRank
3. 基于 cross-encoder 微调得到的任务特定 reranker

### 8.3 我们和训练型 learning-to-rank 的真实关系

最准确的说法不是“我们完全没用训练模型”，而是：

1. 我们的差异化改进部分不是训练出来的，而是启发式融合。
2. 我们仍然使用了一个公开训练好的 reranker 作为精排器。
3. 但我们没有用当前 benchmark 的 qrels 去重新训练一个新的排序模型。

因此，这次实验的技术定位应该表述为：

- 在公开训练好的 reranker 之上，研究启发式检索融合策略如何提升主流 RAG 检索链路的排序质量

### 8.4 什么时候该用启发式，什么时候该用训练型

如果你的目标是：

1. 快速验证系统改进
2. 强调可解释性
3. 没有大量高质量标注训练数据

那启发式融合更合适。

如果你的目标是：

1. 进一步逼近排序上限
2. 有稳定标注数据
3. 能承担训练、验证和过拟合控制成本

那训练型 learning-to-rank 更值得做。

对你这次复试项目来说，当前先用公开 benchmark 做出一套可解释、可复现、可讲清楚的启发式检索改进，是更稳的路线。

## 9. 这套方法如何落到实际 backend RAG

这一步很重要，因为老师很容易追问：

- “你这是离线 benchmark，还是已经接到系统里了？”

当前答案是：

1. 已经接到实际 backend RAG 主链路里了。
2. 不是只留在 `experiments/` 目录里做离线跑分。

实际落地位置主要包括：

1. [hybrid_retrieval.py](/root/Velo/backend/app/services/ai/hybrid_retrieval.py)
   新增文档级 BM25 索引、查询自适应权重、RRF、自适应融合和混合候选池逻辑。
2. [agent_service.py](/root/Velo/backend/app/services/ai/agent_service.py)
   `rag_qa()` 已改成 `向量召回 + 文档级 BM25 + Adaptive Hybrid + Rerank` 主链路。
3. [rerank_service.py](/root/Velo/backend/app/services/ai/rerank_service.py)
   保留 cross-encoder 精排，但把过去过重的标识符硬惩罚收敛成轻量调整，和消融结论保持一致。
4. [config.py](/root/Velo/backend/app/core/config.py)
   新增并启用 `RAG_VECTOR_SEARCH_LIMIT`、`RAG_BM25_SEARCH_LIMIT`、`RAG_HYBRID_CANDIDATE_LIMIT` 等配置。

现在 backend 的实际流程可以概括成：

1. Chroma 先做向量召回。
2. 后端维护一份文档级 BM25 索引做词法召回。
3. 用查询画像动态设置 lexical / dense 权重。
4. 在文档级候选上做自适应融合与混合候选池构造。
5. 再把候选送进 `BAAI/bge-reranker-v2-m3` 做精排。

这里也要讲清楚一个细节：

1. 实验里的“标识符约束”在完整方案里真实存在。
2. 但根据消融结果，当前 backend 落地版本没有再把它当成硬规则，而是只保留轻量影响。
3. 这说明我们不是机械复刻实验公式，而是根据公开 benchmark 证据把系统版调成了更稳的实现。

## 10. 这份结果在复试里怎么讲更稳

建议按下面这个顺序讲：

1. 我先选了公开中文 retrieval benchmark，而不是自造小数据，保证实验可复现。
2. 我把对比对象设成两条主流 RAG 检索链路，而不是很弱的传统流程。
3. 我的改进重点不是提示词，而是检索层的查询自适应融合和多信号粗排。
4. 我还单独做了 DuRetrieval 去模块消融，确认主要增益来自查询自适应权重和混合候选池，而不是随便堆很多技巧。
5. 我给出了严格的排序指标、时延统计和 95% 置信区间，因此结果不只是“看起来更好”，而是“可解释、可核对、可复现”。
6. 最终结果表明，我们的方法在主流混合检索强基线之上仍取得稳定增益，同时代价也被明确量化。

## 11. 边界与下一步

这部分必须老实写，不然老师一追问就容易失分。

1. 当前方法仍然是启发式融合，不是训练型 learning-to-rank。
2. 这次正式 benchmark 评测的是检索质量，不直接等价于最终答案质量。
3. DuRetrieval 去模块消融已经补完，但当前仍只在一套公开数据上做了模块归因，还没有在 T2Retrieval 或私有知识库上复验。
4. 标识符约束在当前公开数据上没有显示稳定正收益，因此更适合被表述为工程假设，而不是已经被公开 benchmark 充分证实的通用结论。
5. 当前结果给出了 bootstrap 置信区间，但还没有补 paired randomization test 这类更正式的显著性检验。
6. 后续如果继续完善，可以补答案忠实度、引用正确率和中文回答一致性评测。
