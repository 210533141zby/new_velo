这部分写的是这次实验在仓库里真实做过的事，不是论文式的“理想流程”。

### 2.1 先做了什么清理

1. 旧的弱基线实验和合成实验目录不再作为主要结论来源，已经从 `experiments/` 中移除。

之前如果拿“传统 `chunk + rerank`”去对比，很容易出现非常夸张的提升，这种结论不适合放到简历里，原因有三点：

1. 基线过弱，他会怀疑是不是故意挑了一个容易赢的对手。
2. 数据过小或过合成时，很容易把启发式规则做成“记忆题”。
3. 如果只比一个非常老的流程，就很难说明你的方案相对当前主流工业检索链路到底强在哪里。

所以这次重做的核心策略是：

1. 用公开中文检索 benchmark，而不是自己杜撰主结论数据。
2. 先对齐主流 RAG 检索方案，再比较我们的方法。
3. 把改动放在检索算法层，而不是只堆提示词。

## 4. 这次到底用了什么数据

### 4.1 数据来源

这次实验使用的是 C-MTEB 公开中文检索基准中的两个 retrieval 数据集：

1. `C-MTEB/DuRetrieval`
2. `C-MTEB/T2Retrieval`

选择它们的原因：

1. 都是公开数据，别人可以复现，不是我自己拍脑袋造题。
2. 都是中文检索任务，和你的中文 RAG 项目更贴近。
3. 规模已经足够大，能避免“小数据偶然性太强”的问题。

### 4.2 数据规模

在本地实际读取时，脚本看到的数据规模如下：

- `DuRetrieval`
  语料库约 `100001` 篇文档，查询约 `2000` 条，qrels 约 `9839` 条。
- `T2Retrieval`
  语料库约 `118605` 篇文档，查询约 `22812` 条，qrels 约 `118932` 条。

qrels 的意思是“标准相关性标注”，也就是每个查询对应哪些文档算正确答案。

### 4.3 parquet 里面是什么结构

脚本实际读到的字段是：

- corpus:
  - `id`
  - `text`
- queries:
  - `id`
  - `text`
- qrels:
  - `qid`
  - `pid`
  - `score`

这里你要记住：

1. `corpus` 是候选文档库。
2. `queries` 是用户问题集合。
3. `qrels` 是标准答案映射，告诉我们某个查询对应哪些文档是正样本。

## 5. 数据是怎么进入脚本的

### 5.1 下载方式

脚本通过 Hugging Face 数据集直链下载 parquet 文件，然后缓存到：

- [public_benchmarks](/root/Velo/experiments/.cache/public_benchmarks)

相关逻辑在 [mainstream_rag_benchmark.py](/root/Velo/experiments/mainstream_rag_benchmark.py) 的 `load_public_dataset()` 中。

### 5.2 文本清洗

每条文档文本和查询文本进入检索前，会先做 `clean_text()` 处理，主要包括：

1. HTML 实体反转义。
2. `<br>` 转成空格或换行语义。
3. 去掉 HTML 标签。
4. 合并多余空白字符。
5. 空文本回退为 `空白内容`。
6. 文档文本默认截断到最多 `1600` 字符，避免超长文本拖慢编码。

你要理解这一点：

这不是“随便截断”，而是工程上常见的折中。因为公开数据里有大量带图片标签、步骤截图、网页残留格式的长文档，不先规整会直接影响 embedding 服务稳定性和 rerank 时延。

### 5.3 分词与词项标准化

BM25 这一支需要词法分词，所以脚本用了 `jieba` 分词，并且做了简单的标准化：

1. 中文词保留。
2. 英文和数字拆分并转小写。
3. 混合 token 尽量拆成更稳定的字母段和数字段。

这部分逻辑在 `tokenize_text()`。

### 5.4 为什么还要抽样查询

`T2Retrieval` 查询很多，`DuRetrieval` 也不小。为了让单轮 benchmark 在本机可控时间内跑完，脚本默认每个数据集抽样固定数量查询做评测。

当前脚本默认配置是：

- 每个数据集最多抽样 `1200` 条查询
- 随机种子固定为 `20260415`

这样做的目的不是偷懒，而是：

1. 保证每次复现实验拿到同一批查询。
2. 把计算预算集中在“公开 benchmark 上的稳定对比”，而不是把时间消耗在无意义的重复计算上。

## 6. 主实验对比的是哪三条链路

这次不是只比一个“老式方案”。

我们对比三条更接近真实工业检索系统的链路：

### 6.1 主流 RAG 方案 A

`Dense Retrieval + Rerank`

流程是：

1. 查询做 embedding。
2. 对整个文档库做向量相似度召回。
3. 取前 `Top 20` 候选。
4. 用 cross-encoder reranker 精排。
5. 返回前 `Top 10`。

这代表很多系统最容易落地的单路语义检索方案。

### 6.2 主流 RAG 方案 B

`Hybrid Retrieval + RRF + Rerank`

流程是：

1. Dense 检索召回一份候选。
2. BM25 检索再召回一份候选。
3. 用 RRF 融合两份排序结果。
4. 取融合后的前 `Top 20` 候选。
5. 用 cross-encoder reranker 精排。
6. 返回前 `Top 10`。

这更接近今天很多搜索和 RAG 系统的主流做法。

### 6.3 我们的方案

`Adaptive Hybrid + Multi-signal Fusion + Rerank`

在主流混合检索的基础上，再做三层增强：

1. 查询自适应权重
   不是对所有问题固定使用同一组 dense/BM25 权重，而是根据查询长度、是否含编号、词项稀有度动态调节。
2. 多信号融合
   除了 dense 分数和 BM25 分数，还把 RRF、词法覆盖率、标识符一致性一起纳入候选打分。
3. 更大的精排候选池
   主流方案只把前 `20` 个候选送入 rerank，我们的方案会构造最多 `30` 个更干净的候选池再精排。

## 7. 三种算法到底怎么实现

这一节最关键，你答辩时主要就讲这里。

### 7.1 Dense Retrieval

Dense retrieval 的思想是把查询和文档都编码成向量，然后用向量相似度做召回。

在这次脚本里：

1. 文档和查询先做 embedding。
2. embedding 做 L2 归一化。
3. 归一化后直接做点积，相当于 cosine similarity。
4. 每个查询取相似度前 `Top 50` 作为 dense 候选。

正式实验当前使用的是本地可稳定运行的 embedding 服务：

- `nomic-embed-text:latest`

原因非常现实：

1. 它能直接通过本机 Ollama embedding API 稳定跑通大规模语料。
2. 这次 benchmark 的重点是检索链路设计，而不是单个 embedding 模型的 leaderboard 刷分。
3. rerank 阶段仍然使用了更强的 `BAAI/bge-reranker-v2-m3`。

后续如果本地 `BAAI/bge-m3` 直载版本稳定，也可以直接替换进去，脚本已经支持显式指定 embedding model。

### 7.2 BM25

BM25 是经典词法检索方法，本质上是：

1. 看查询词有没有出现在文档里。
2. 词在文档里出现得多不多。
3. 词在整个语料里是不是稀有词。
4. 同时对文档长度做归一化，防止长文天然占便宜。

这一步的意义很大，因为很多中文问题里仍然存在：

1. 专有名词
2. 数字编号
3. 产品型号
4. 精确关键词

这些信息向量模型不一定稳定，但 BM25 往往能抓得很准。

### 7.3 RRF 融合

RRF 全称是 Reciprocal Rank Fusion。

它不直接混合“原始分数”，而是混合“排序名次”。公式可以写成：

```text
RRF(d) = Σ 1 / (k + rank_i(d))
```

这里：

- `d` 是某篇文档
- `rank_i(d)` 是它在第 `i` 个检索列表中的名次
- `k` 是平滑常数，这次脚本里固定为 `60`

为什么主流工业方案喜欢它：

1. Dense 分数和 BM25 分数的量纲不同，直接加权很麻烦。
2. RRF 不需要强制做复杂的分数校准。
3. 它对“在多个列表里都排得比较靠前”的文档特别友好。

### 7.4 查询自适应权重

这是我们的关键改动之一。

脚本会先为每个查询算一个 profile，核心字段是：

1. `token_count`
2. `has_identifier`
3. `avg_idf`
4. `max_idf`

然后按启发式规则生成 `lexical_weight`，再令：

```text
dense_weight = 1 - lexical_weight
```

当前代码中的规则是：

1. 初始 `lexical_weight = 0.36`
2. 如果查询里有字母或数字标识符，`+0.18`
3. 如果查询很短，`+0.14`
4. 如果查询很长，`-0.08`
5. 如果存在高 idf 稀有词，再 `+0.10`
6. 最后裁剪到 `[0.24, 0.78]`

你可以这样理解：

1. 短问题、带编号问题、带关键词问题，更偏向 BM25。
2. 长问题、抽象表述问题，更偏向 dense。

这比“所有问题一套固定权重”更接近真实用户查询。

### 7.5 词法覆盖奖励

我们额外定义了：

```text
coverage_ratio = |query_tokens ∩ doc_tokens| / |query_tokens|
```

它表示查询词里有多少比例在文档中被覆盖。

这一步的作用是：

1. 防止 dense 检索召回“语义很像但关键词并不对”的文档。
2. 让真正包含核心词的候选更容易进入 rerank。

### 7.6 标识符一致性约束

对于包含数字、型号、带编号字符串的查询，脚本会额外提取标识符集合，比如：

1. `3090`
2. `v2`
3. `cuda12.6`
4. `abc-123`

然后检查文档里有没有同类标识符重合。

如果查询里本来就有标识符，而文档完全没有对应重合，脚本会给一个负向惩罚。

这一步特别适合：

1. 带版本号的问题
2. 带课程编号、型号编号的问题
3. 带年份和数值的问题

但这里要和最终口径区分开：

1. 它属于完整方案里真实实现过的一部分。
2. 不过从后面补做的 `DuRetrieval` 去模块消融看，这一项在当前公开 benchmark 上没有显示稳定正收益。
3. 所以答辩时可以说“我尝试过这个工程信号”，但不要把它讲成已经被公开数据稳定验证的核心贡献。

### 7.7 我们最终的候选融合分数

在我们的方案里，某篇候选文档的融合分数是多信号组合出来的：

```text
adaptive_score
= dense_weight * normalized_dense
+ lexical_weight * normalized_bm25
+ 0.26 * normalized_rrf
+ 0.08 * coverage
+ 0.06 * identifier_overlap
```

如果查询包含标识符，但该候选的 `identifier_overlap == 0`，再额外减去 `0.10`。

你要注意两点：

1. 这不是训练出来的学习排序模型，而是公开 benchmark 上可解释、可复现的启发式融合策略。
2. 它的价值在于“解释性强、工程实现成本低、效果通常比固定权重更稳”。

### 7.8 候选池怎么送入 rerank

我们的方案不会只看一份排序列表，而是构造一个更稳妥的候选池：

1. 先取自适应排序前半部分。
2. 再补一部分 RRF 排名靠前的文档。
3. 如果还不够，再继续按自适应排序补齐。
4. 最终最多送 `30` 个候选进入 rerank。

这相当于：

1. 用我们的融合分数做“精一点的粗筛”
2. 再让 cross-encoder 做“贵一点但更准的精筛”

## 8. 为什么还要 rerank

因为前面的 dense、BM25、RRF、本质上都还属于召回或粗排阶段。

rerank 的作用是：

1. 让模型真正同时看到“查询 + 文档”这一对输入。
2. 直接判断二者相关性，而不是只看各自独立编码后的向量空间距离。

当前使用的精排模型是：

- `BAAI/bge-reranker-v2-m3`

脚本里通过 `CrossEncoder` 加载本地模型目录：

- [/root/Velo/backend/data/models/rerank/BAAI--bge-reranker-v2-m3](/root/Velo/backend/data/models/rerank/BAAI--bge-reranker-v2-m3)

rerank 时每个候选只截取文档前 `900` 字符，主要是为了控制 cross-encoder 时延。

## 9. 指标是怎么算的

这次实验不是只看一个命中率。

### 9.1 Hit@1

看排序第一的文档是不是正确文档。

意义：

如果你希望最终给大模型的上下文尽量短、尽量准，Hit@1 非常重要。

### 9.2 MRR@10

看第一个正确文档出现得有多靠前。

如果正确文档排在第 1 名，贡献是 `1.0`；排在第 2 名，贡献是 `0.5`；越往后越低。

意义：

它能反映“第一个有效证据”来得早不早。

### 9.3 Recall@10

看前 `10` 个结果里覆盖了多少正确文档。

意义：

如果你的生成阶段会喂多个引用，Recall@10 很重要。

### 9.4 nDCG@10

它既考虑有没有召回到正确文档，也考虑正确文档排得靠不靠前。

意义：

这是排序任务里非常常见的综合指标，比只看 Hit@1 更全面。

### 9.5 平均时延

脚本会记录每个查询在各条 pipeline 上的平均耗时，单位是毫秒。

意义：

你答辩时不能只讲“效果更好”，还要讲“代价有没有明显增加”。

### 9.6 95% Bootstrap 置信区间

这次脚本额外做了 `bootstrap mean CI`，用于给 `Hit@1` 和 `nDCG@10` 生成 95% 置信区间。

这么做的原因是：

1. 单次平均值有时容易被误解成“精确不变”。
2. 有了区间后，结果会显得更像一个严肃 benchmark，而不是只报一个点估计。

## 10. 脚本是怎么保证可复现的

主要有四个手段：

1. 固定随机种子 `20260415`
2. 查询抽样固定
3. embedding 结果按数据集和 id 签名缓存
4. 输出完整保存为 CSV、JSON、SVG 和 Notebook

其中 embedding 缓存命名会包含：

1. 数据集名
2. embedding 模型名
3. 当前文档或查询 id 集合的签名

这样做能避免：

1. smoke 测试缓存污染正式实验
2. 换了数据子集却误用旧向量

## 11. 这次为了稳定性做了哪些工程处理

这是你以后做项目时很有用的一节，因为真实系统里“算法正确”还不够，还要“跑得稳”。

### 11.1 embedding 失败回退

脚本里的 `embed_texts()` 做了多级回退：

1. 正常文本直接请求 embedding API
2. 失败后缩短文本
3. 再失败就做字符清洗
4. 单条文本仍失败时，用 `空白内容` 的 embedding 兜底

这么做不是为了美化结果，而是为了防止：

公开网页语料中个别脏文本把整轮实验跑挂。

### 11.2 长跑进度输出

脚本现在会打印：

1. 当前使用的 embedding model
2. 当前数据集规模
3. doc embedding 是否命中缓存
4. query embedding 是否命中缓存
5. 查询处理进度

这样你后续自己跑时不会遇到“卡着不动不知道在干嘛”的问题。

### 11.3 阶段性结果落盘

在继续正式实验时，脚本又补了一个重要的工程改动：

1. 每完成一个数据集，就先把当前阶段结果写入 `outputs/`
2. 不再等所有数据集都结束后再统一写盘

这一步的意义不是提升检索指标，而是提升实验可靠性：

1. 如果第二个数据集在 embedding 服务阶段出故障，第一个数据集的正式结果不会丢
2. benchmark 变得更适合长时间运行和中断恢复
3. 实验输出更接近真正研究项目要求的可恢复性

## 12. 我们借鉴了哪些论文和工业方案

这一节很重要，因为答辩老师很可能会问“你的方法依据是什么”。

### 12.1 数据基准来源

借鉴来源：

- C-Pack / C-MTEB

它给我们提供了：

1. 中文 embedding benchmark 的任务划分
2. 公开可复现的 retrieval 数据集
3. 统一的中文检索评测视角

### 12.2 混合检索思路来源

借鉴来源：

- BGE-M3 模型卡和论文
- Azure AI Search 的 hybrid search 官方文档
- Elastic 的 hybrid search 官方资料

共同结论是：

1. 只做 dense retrieval 不够稳
2. 混合 lexical + dense 是更主流的工程实践
3. rerank 常常要放在混合召回之后

### 12.3 重排思路来源

借鉴来源：

- BAAI/bge-reranker-v2-m3 模型卡
- Elastic semantic reranking 官方文档

核心思想：

1. 先做廉价召回
2. 再对 top-k 用 cross-encoder 做高精度精排

### 12.4 RRF 来源

借鉴来源：

- Cormack, Clarke, Buettcher 在 SIGIR 2009 提出的 Reciprocal Rank Fusion

工程上我们同时参考了：

- Azure AI Search 的 RRF 说明
- Elastic 对 RRF 与 hybrid search 的解释

### 12.5 我们没有照搬什么

这次实验没有直接上：

1. 多轮 Self-RAG
2. GraphRAG
3. 训练型 Learning to Rank
4. LLM 生成式 query rewriting

原因不是这些方向没价值，而是：

1. 当前先把“单轮检索链路”做扎实，更容易讲清楚。
2. 对毕业设计和复试来说，先把主链路 benchmark 做干净，比一次堆太多概念更稳。

### 15.2 消融实验是怎么做的，为什么重做协议

这里要特别注意：

1. 最终保留的不是“累加式消融”。
2. 因为前一版累加式设计会把“候选池大小变化”和“模块本身作用”混在一起。
3. 所以最后重做成了更稳的“去模块消融”协议。

最终协议是：

1. 以完整方案为中心
2. 每次只去掉一个模块
3. 其他设置尽量保持不变
4. 另外保留 `主流方案 B` 作为外部参考线

也就是说，最后真正回答的是：

- “如果把完整方案里的某一个模块拿掉，性能会掉多少”

而不是：

- “从弱版本一路往上堆模块后，最后总共涨了多少”

### 15.3 这轮消融具体比较了什么

消融实验当前只在 `DuRetrieval` 上做，使用与主实验一致的固定抽样：

1. 数据集：`DuRetrieval`
2. 查询数：`1200`
3. dense / BM25 初始召回深度：`Top 50`
4. 完整方案 rerank 候选池：`Top 30`
5. 主流方案 B rerank 候选池：`Top 20`
6. bootstrap 重采样次数：`1000`

比较的配置包括：

1. `主流方案 B`
2. `去掉自适应权重`
3. `去掉覆盖率奖励`
4. `去掉标识符约束`
5. `去掉混合候选池`
6. `完整方案`

### 15.4 去模块消融的核心结果是什么

这轮消融最值得看的是 `Hit@1` 和 `nDCG@10`，核心结果如下：

| 配置 | Hit@1 | nDCG@10 | 相对完整方案的 Hit@1 变化 | 相对完整方案的 nDCG@10 变化 |
| --- | --- | --- | --- | --- |
| 主流方案 B | 0.8408 | 0.6616 | -0.0150 | -0.0242 |
| 去掉自适应权重 | 0.8392 | 0.6558 | -0.0167 | -0.0300 |
| 去掉覆盖率奖励 | 0.8542 | 0.6820 | -0.0017 | -0.0038 |
| 去掉标识符约束 | 0.8567 | 0.6874 | +0.0008 | +0.0017 |
| 去掉混合候选池 | 0.8342 | 0.6323 | -0.0217 | -0.0535 |
| 完整方案 | 0.8558 | 0.6858 | 0.0000 | 0.0000 |

这里的意思不是“所有模块都有效”，而是：

1. 去掉查询自适应权重后，指标明显下降，说明它是主要贡献源之一。
2. 去掉混合候选池后，下降最大，说明它是当前公开数据上最强的贡献项。
3. 去掉覆盖率奖励后，仍有小幅下降，说明它更像补充收益。
4. 去掉标识符约束后，均值没有变差，说明这项设计在当前公开 benchmark 上没有显示稳定正贡献。

### 15.5 这轮消融最后应该怎么讲才稳

最稳的表达不是：

- “我的所有模块都被公开 benchmark 证明有效”

而是：

1. 主实验证明完整方案相对主流混合检索强基线仍有稳定提升。
2. 去模块消融进一步说明，主要增益来自查询自适应权重和混合候选池。
3. 覆盖率奖励提供小幅补充收益。
4. 标识符约束属于真实实现过的工程信号，但当前 `DuRetrieval` 公开数据还不支持把它讲成稳定正贡献。

这样讲的好处是：

1. 更真实
2. 更经得起追问
3. 更像认真做过检索实验，而不是只挑好看的数字说

## 18. 真实执行日志与小动作记录

这一节专门记录这次实验在真实执行中发生了什么，不再只写“理想流程”，而是写：

1. 我看到了什么现象
2. 我据此做了什么判断
3. 我接着做了什么动作

这样做的好处是，你以后答辩时不会只会背算法定义，而是能讲出一套真正做过实验的人会讲的话。

### 18.1 为什么我要一直盯进度，而不是只会等结果

长时间 benchmark 最怕两件事：

1. 程序其实已经卡死，但日志不明显。
2. 程序没有卡死，只是在做重计算，用户却以为出问题了。

所以我在跑实验时会反复观察：

1. 有没有命中 embedding 缓存
2. GPU 利用率是不是还在工作
3. 当前是在向量化、BM25 建索引，还是在 rerank
4. 输出目录里有没有新文件落盘

这不是“多余动作”，而是为了区分“正常慢”和“异常挂”。

### 18.2 真实恢复点判断

第一次恢复实验前，我先做的不是直接重跑，而是先核对：

1. `DuRetrieval` 的 doc embedding 缓存是否存在
2. `DuRetrieval` 的 query embedding 缓存是否存在
3. 当前是不是还有旧的 benchmark 进程残留

判断依据是：

1. 如果缓存没了，说明不能说“从昨天继续”
2. 如果有旧进程残留，可能会覆盖今天的正式输出

所以当时我的动作是：

1. 先读取当时写下的暂停恢复备注
2. 再检查缓存文件是否还在
3. 最后确认当前没有旧的正式 benchmark 进程在跑

### 18.3 为什么我会特别强调“命中缓存”

正式恢复后，我专门去看日志里有没有出现：

- `reuse cached doc embeddings`
- `reuse cached query embeddings`

因为这两行非常关键。

如果它们出现，说明：

1. `DuRetrieval` 不会重新做最重的向量化
2. 这次重跑是有效续跑，不是白跑一遍

所以我当时的判断是：

“只要这两份缓存复用上，今天的主要时间就会花在 rerank 评测和第二套数据集上，而不是回到昨天已经做完的重活。”

### 18.4 为什么我会去看 GPU 利用率

在日志短时间不刷新的时候，我额外看了 GPU 利用率。

这是因为：

1. 如果 GPU 接近满载，通常说明 dense 排序、embedding 或 rerank 正在正常跑。
2. 如果 GPU 是空的，而程序又没有新日志，就要怀疑是不是停在 CPU 分词、索引构建，或者已经挂掉。

所以当时当我看到 GPU 利用率接近满载时，我给出的判断不是“完全没问题”，而是更准确的一句：

“程序大概率在正常推进，不是在空等。”

这类表述很重要，因为它既真实，又不会过度承诺。

### 18.5 为什么我会一直报 `100/1200`、`300/1200` 这种进度

在 `DuRetrieval` 的正式评测阶段，日志逐步出现：

1. `processed 100/1200`
2. `processed 200/1200`
3. `processed 300/1200`
4. 一直到 `1100/1200`

这里我做的不是简单复读，而是在每个阶段都做了一个外部判断：

1. `100/1200`：说明已经进入逐查询正式评测，主链路稳定。
2. `300/1200`：说明速度比较稳定，可以初步估算剩余时间。
3. `700/1200`：说明已经超过昨天的暂停点，今天产生了新的净进度。
4. `1100/1200`：说明 `DuRetrieval` 快收尾，可以提前准备关注 `T2Retrieval` 是否能顺利起跑。

也就是说，这些小进度提示的价值不只是“报数”，而是帮助判断下一步风险点在哪里。

### 18.6 为什么 `DuRetrieval` 跑完后，我立刻把注意力转到 `T2Retrieval`

当 `DuRetrieval` 完成后，我立刻关注的不是先庆祝，而是：

1. `T2Retrieval` 是否会复用现成缓存
2. 如果没有缓存，它是否能顺利开始首轮 doc embedding

原因很简单：

1. 第一套数据已经证明评测链路能跑
2. 第二套数据才是新的不确定性来源

所以我当时的外部判断是：

“`DuRetrieval` 的问题已经转化为已知问题，接下来真正要盯的是 `T2Retrieval` 的首轮文档向量化是否稳。”

### 18.7 为什么出现“没日志但我说是正常”的情况

在切换到 `T2Retrieval` 之后，有一段时间没有新日志。

这时我没有立刻判定为失败，而是先做了两个小动作：

1. 看 GPU 利用率
2. 看缓存目录里有没有 `t2_*` 新文件

当时我的判断逻辑是：

1. 如果 GPU 低、没有新缓存，但进程还活着，大概率在做 CPU 侧分词、BM25 建索引或前置准备。
2. 这属于“慢”，但还不能立刻定性为“挂”。

这类判断很像真实做系统实验时的状态，而不是只会看最终报错。

### 18.8 真正出故障时，我是怎么判断不是算法错了

后面 `T2Retrieval` 在开始做 doc embedding 时，程序报错：

- `Connection refused`
- 地址是 `127.0.0.1:11434`

我当时首先判断的不是“算法坏了”，而是“embedding 服务掉了”，依据有三个：

1. 报错发生在 HTTP embedding 请求阶段
2. 失败地址是本地 Ollama API
3. 错误类型是连接拒绝，而不是模型输出异常

所以当时的结论是：

1. 不是 dense / BM25 / rerank 公式有问题
2. 不是数据集本身损坏
3. 是本地 embedding 服务中途退出

这类区分在答辩里也很重要，因为它体现你知道“算法错误”和“运行时基础设施故障”不是一回事。

### 18.9 出故障后我具体做了哪些小动作

不是看到报错就结束，而是按下面顺序处理：

1. 检查 `ollama serve` 是否还在进程列表里
2. 用 `curl http://127.0.0.1:11434/api/tags` 验证接口是否真的不可达
3. 确认 `t2` 缓存尚未生成，避免误判为半成品可用
4. 重新启动 `ollama serve`
5. 再次请求 `api/tags`，确认 embedding 服务恢复

这些动作看起来很小，但它们组成了一套非常真实的排障闭环：

1. 先验证故障
2. 再恢复服务
3. 最后确认恢复成功

### 18.10 为什么我又顺手改了脚本

当时我没有只做“手动重启服务”，还顺手补了一个脚本增强方向：

“每完成一个数据集，就先把阶段性结果落盘。”

原因很现实：

1. 这次 `DuRetrieval` 已经完整跑完
2. 但因为脚本原本是整轮结束后才统一写文件，所以 `T2Retrieval` 一挂，前面的结果也没正式保存

这不是理论改进，但非常重要，因为它体现了真正做长跑实验时的工程意识：

1. 结果要可恢复
2. 失败不能让前功尽弃

### 18.11 这些“小动作”为什么值得写进文档

因为真正能打动老师的，往往不是你背了多少术语，而是你能否表现出下面这些能力：

1. 你知道实验慢的时候应该看什么信号
2. 你知道怎么区分缓存复用、CPU 预处理、GPU 计算和服务崩溃
3. 你知道出现问题时，先验证环境，再判断算法
4. 你会顺手把一次真实故障总结成下一次更稳的脚本设计

这就是“研究意识”和“系统实现能力”的结合点。

### 18.12 你答辩时可以怎么讲这部分

你可以这样讲：

“这次实验不是一键跑出来的。我在正式 benchmark 过程中，持续观察了缓存命中、GPU 利用率、查询处理进度和 embedding 服务状态。比如在 `DuRetrieval` 恢复时，我先确认 doc 和 query embedding 缓存已经落盘，再继续正式评测，避免重复向量化；在切换到 `T2Retrieval` 后，我又通过 GPU 利用率和缓存目录判断它是在做前置计算还是已经异常；当 Ollama embedding 服务中途退出时，我根据 `Connection refused` 很快区分出这是基础设施故障而不是算法错误，然后重启服务并补强脚本的阶段性落盘能力。这部分经历让我对 RAG 实验不只是停留在算法定义上，而是真正理解了一套检索 benchmark 如何稳定落地。” 

## 19. 这轮实验目前已经完成到哪里

如果按“能不能拿去复试讲”和“有没有真正落地”这两个标准看，这轮已经完成了四件关键事情：

1. 主实验已经完整跑完。
   两套公开中文检索 benchmark 都有正式结果，不再停留在 smoke 或半成品阶段。
2. 去模块消融已经补完。
   `DuRetrieval` 上已经回答了“到底是哪几个模块在起主要作用”，最稳的结论是查询自适应权重和混合候选池贡献最大。
3. 负结果已经被诚实保留。
   标识符约束在当前公开 benchmark 上没有显示稳定正收益，因此不会被硬包装成核心贡献。
4. 这套方法已经接入实际 backend RAG。
   也就是说，这不再只是 `experiments/` 里的离线 benchmark，而是已经落到系统检索主链路里。

## 20. 如果后续还要继续做，最值得补什么

这一轮不是没东西做了，而是主线已经完成，后续可以更聚焦：

1. 在项目私有知识库上补人工标注问答集，验证公开 benchmark 上的结论能不能迁移到真实语料。
2. 加入答案级评估，例如忠实度、引用正确率、中文回答一致性。
3. 把去模块消融从 `DuRetrieval` 扩到 `T2Retrieval` 或私有知识库，补强模块归因的外部有效性。
4. 如果后面真想继续冲上限，再考虑 query rewriting、训练型 learning-to-rank、GraphRAG 或多轮 Self-RAG。
