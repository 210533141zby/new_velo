# 复试答辩话术

这份材料不是论文正文，而是给复试现场直接用的话术稿。

## 1. 30 秒版

“我这部分工作的重点不在提示词，而在 RAG 的检索链路优化。我选用了 C-MTEB 里的两套公开中文 retrieval benchmark，先对齐了两条主流方案：`Dense + Rerank` 和 `Hybrid + RRF + Rerank`。然后我在混合检索上加入查询自适应权重、覆盖率奖励和更稳的混合候选池，最终在两套公开数据上相对主流混合检索强基线仍取得了稳定提升，并且已经把这套检索策略接入了项目 backend 的实际 RAG 主链路。” 

## 2. 1 分钟版

“我这次 RAG 改进的核心不在生成提示词，而在检索算法层。为了避免用弱基线刷出夸张数字，我直接选了 C-MTEB 的 `DuRetrieval` 和 `T2Retrieval` 两套公开中文检索 benchmark，先搭了两条主流对照链路：第一条是 `Dense Retrieval + Rerank`，第二条是更接近工业系统的 `Hybrid Retrieval + RRF + Rerank`。在这个强基线之上，我做了查询自适应权重、多信号融合和更稳的候选池设计。正式结果显示，相对主流混合检索强基线，我的方法在 `Hit@1`、`MRR@10`、`Recall@10` 和 `nDCG@10` 上都保持正提升。我还补了去模块消融，确认主要收益来自查询自适应权重和混合候选池，而不是简单堆技巧。最后，我把这套检索策略接进了 backend 实际 RAG，而不是只停留在离线实验里。” 

## 3. 3 分钟版

“我做的不是一个只调提示词的 RAG 项目，而是一套检索链路优化实验。最开始我发现，如果拿很弱的传统 `chunk + rerank` 当对照，很容易得到夸张提升，但这种结果不适合写进复试简历，因为老师会怀疑是不是故意挑了好赢的对手。所以我把实验完全重做，换成 C-MTEB 的公开中文 retrieval benchmark，包括 `DuRetrieval` 和 `T2Retrieval`。

在实验设计上，我先对齐了两条主流链路。方案 A 是 `Dense Retrieval + Rerank`，方案 B 是 `Hybrid Retrieval + RRF + Rerank`。然后在方案 B 的基础上，我加入了三类改进：第一是查询自适应权重，也就是短查询、带编号查询更偏 BM25，长查询、抽象查询再更多依赖 dense；第二是多信号粗排，不只看 dense 和 BM25，还加入覆盖率奖励；第三是更稳的混合候选池，让 reranker 不至于只在一份过窄候选上排序。

指标上，我主要报告 `Hit@1`、`MRR@10`、`Recall@10`、`nDCG@10` 和平均时延。正式结果显示，相对主流混合检索强基线，我的方法在两套公开中文数据上都保持正提升。然后我又单独做了 DuRetrieval 去模块消融，结论是主要收益来自查询自适应权重和混合候选池，覆盖率奖励有小幅补充收益，而标识符约束在当前公开数据上没有显示稳定正收益，所以我没有把它硬讲成核心贡献。

另外，这项工作不是只停在 `experiments/` 目录里。我已经把这套 `Adaptive Hybrid + Rerank` 的思路接入了项目 backend 的实际 RAG 主链路，用文档级 BM25、查询画像和混合候选池替代了原来更粗糙的 heuristic 检索流程。所以这部分既有公开 benchmark 支撑，也有系统落地。” 

## 4. 常见追问

### 4.1 为什么选这两个数据集

“因为它们是公开中文 retrieval benchmark，能复现，也更贴近中文 RAG 项目。这样做比自造小数据更稳，也更能避免实验设计偏向自己方案。” 

### 4.2 你的提升到底怎么算

“我先对每条查询分别计算 `Hit@1`、`MRR@10`、`Recall@10`、`nDCG@10`，再对数据集内全部查询取平均，得到数据集级结果。相对主流方案 B 的提升，默认报告的是绝对增量，比如 `0.8558 - 0.8408 = 0.0150`，也就是 `Hit@1` 提升 `1.50` 个百分点。” 

### 4.3 为什么说你这个不是训练型 learning-to-rank

“因为我这次改进的核心部分没有用 qrels 去训练排序模型，也没有做梯度更新。查询画像规则、融合权重和候选池策略都是启发式设计。整条链路里用到的 cross-encoder reranker 本身是公开训练好的模型，但我没有在当前 benchmark 上重新训练它。” 

### 4.4 标识符约束为什么没有继续讲成核心贡献

“因为我补了去模块消融之后，发现它在当前 `DuRetrieval` 公开 benchmark 上没有显示稳定正收益。如果我还把它讲成核心贡献，老师一追问就站不住。所以我现在把它更准确地表述成：一个面向真实企业知识库编号查询的工程假设，但在当前公开数据上还需要进一步验证。” 

### 4.5 这项工作有没有落到系统里

“有。我已经把实验里的 `Adaptive Hybrid + Rerank` 思路接进 backend 的实际 RAG 主链路，新增了文档级 BM25、查询自适应权重、RRF 和混合候选池，不再只是离线脚本结果。” 

## 5. 现场表达注意点

1. 先讲“为什么重做实验”，再讲“怎么改方法”，最后再讲“结果和落地”。
2. 不要把标识符约束讲成已经被公开 benchmark 充分证实的主要贡献。
3. 强调你控制了评测协议一致性，不是换模型、换数据、换设置后硬比结果。
4. 如果老师追问系统实现，就直接说 backend 已经接入文档级 BM25 和自适应 hybrid 检索主链路。
